---
name: mxy-organize-code
description: 为文件的函数/变量/钩子增加注释，并按标准分层重新排序代码，支持单文件和多文件批量处理
---

请对用户指定的文件执行以下步骤。支持三种模式：

- **单文件**：在 IDE 中打开一个文件，或直接指定文件路径
- **多文件**：指定多个文件路径（空格/逗号分隔），或 glob 模式（如 `src/business/admin/**/*.vue`）
- **目录**：指定目录路径，展开为该目录下所有 `.vue`/`.ts`/`.js` 文件（排除 `node_modules`）

## 步骤 1：确定目标文件列表

最终产出是一个 `目标文件列表`（`string[]`），后续步骤对列表中的每个文件顺序执行。

### 1a：收集候选文件

按优先级获取：

1. **用户显式指定的路径**（最高优先级） — 可以是单个文件、空格/逗号分隔的多个文件、glob 模式（如 `src/business/**/*.vue`）、或目录路径
2. **对话上下文中提及的文件路径**
3. **以上两者都没有时**，必须向用户询问目标文件路径，不可自行猜测

### 1b：展开文件列表

如果用户输入了 glob 模式或目录路径，使用 Glob 工具展开为具体文件列表（`glob: "!node_modules/**"` 排除 node_modules）。

展开后如果文件数超过 **15 个**

> 必须调用 `question` 工具提醒用户

**必须调用 `question` 工具**提醒用户当前匹配到的文件数量，选项：是，全部处理 / 否，我要筛选。用户确认后再继续。

### 1c：逐文件类型校验与路由

对每个候选文件检查扩展名：

| 文件类型 | 处理方式 |
|---------|---------|
| `.vue`、`.ts`、`.js` | ✅ 加入待处理列表，继续到步骤 1d |
| `.scss`、`.css` | ⏭️ 标记跳过，原因：应使用 `mxy-organize-scss` 技能 |
| 其他扩展名 | ⏭️ 标记跳过，原因：本技能仅支持 Vue SFC / TS / JS 文件 |

跳过文件不阻塞整体流程，在最终汇总报告中列出。

### 1d：检测 Vue2 混搭模式（仅 `.vue` 文件）

对本技能而言，`.vue` 文件应先升级到纯 Vue3 语法，再进行注释和排序。

对每个 `.vue` 文件快速扫描（Read 前 80 行脚本部分），检查是否存在以下混搭特征：

| 特征 | grep 模式 | 说明 |
|------|----------|------|
| `getCurrentInstance` | `getCurrentInstance` | 通过 proxy 访问全局属性 |
| 双重脚本块 | 存在两个 `<script` 标签 | Options API 名称注册 + Composition API 逻辑分离 |
| `export default {` | `export default \{` | Options API 包装器（`setup()` 或传统 `data/methods`） |
| `defineComponent` 独立调用 | `defineComponent(` | 应在 `<script setup>` 中用 `defineOptions` 替代 |
| `components: {` | `components:\s*\{` | Vue2 局部组件注册 |

**处理策略：**

- **命中任一特征**：提示用户该文件存在 Vue2 混搭语法，建议先运行 `mxy-upgrade-vue3` 升级语法。

> 必须调用 `question` 工具提醒用户

**必须调用 `question` 工具**询问：跳过此文件（推荐）/ 仍然整理。
- **未命中**：文件已是纯 Vue3 语法，进入步骤 2-5 正常处理。
- **检测不确定时**（如文件过大只读取了前 80 行，无法确定是否存在 `export default`）：标注文件为"待确认"，仍然进入处理流程，但在执行步骤 2 时做完整扫描。

跳过文件不阻塞整体流程，在最终汇总报告中列出，原因标注为"存在 Vue2 混搭语法，建议先运行 mxy-upgrade-vue3"。

## 步骤 2-5：逐文件处理

对步骤 1 产出的每个目标文件（`目标文件列表` 中且校验通过的文件），**顺序执行**步骤 2-5。单个文件处理失败不中断整体流程，记录失败原因后继续处理下一个。

处理每个文件前输出进度：`正在整理 [N/总数]: path/to/file.vue`

### 步骤 2：分析当前代码

识别以下代码元素，记录它们的作用和依赖关系：

- **导入声明**：来自哪些模块，是否有多余导入
- **响应式状态**：`ref` / `reactive` / `computed` — 每个变量的用途
- **函数/方法**：签名、用途、关键逻辑中的 `if` 分支原因
- **监听器**：`watch` / `watchEffect` — 监听目标、触发时机
- **生命周期钩子**：`onMounted` / `onUnmounted` 等 — 注册/清理的对称性
- **内部变量**（非响应式）：闭包变量、模块级常量等 — 不与 UI 绑定的原因

## 步骤 3：增加注释

为代码元素添加单行注释，遵循以下规则：

### 注释内容（解释 WHY，不重复 WHAT）

```ts
// ✅ 正确：解释为什么需要这样做
/** 带 token 的 URL 需要 2500ms 遮罩，SSO 登录跳转时间不可控 */
const loadingTime = ref(500);

// ❌ 错误：重复代码本身
/** 设置 loadingTime 为 500 */
const loadingTime = ref(500);
```

### `if` 分支必须注释

```ts
// ✅ 正确：说明进入分支的原因
// URL 参数中携带 token 时需延长 loading，遮盖 SSO 登录跳转过程
if (urlParam && urlParam.includes('token')) {
  loadingTime.value = 2500;
}

// ❌ 错误：重复条件表达式
// 如果 urlParam 存在且包含 token
if (urlParam && urlParam.includes('token')) {
```

### 函数注释（单行 JSDoc）

```ts
/** 等待子页面就绪后发送消息（已就绪立即发送，未就绪排队等待） */
function sendAfterReady(data: any): Promise<boolean> { ... }
```

### 状态变量注释

```ts
/** 子页面是否已发送 CHILD_READY 就绪信号 */
const isChildReady = ref(false);

/** 等待 CHILD_READY 的共享 Promise，多个 sendAfterReady 调用方共用 */
let _readyPromise: Promise<boolean> | null = null;
```

### 注释长度约束

- 所有注释不得超过 100 字符/行（含 `//` 或 `/**` 前缀）
- 函数职责复杂时拆为多行单行注释，不使用多行 `/* */` 块注释

## 步骤 4：重新排序代码

按照文件类型的标准分层重新排列代码块。每个分层之间用分隔注释标注。

### Vue 3 SFC 顶层块排序（硬约束）

Vue 单文件组件（`.vue`）的顶层块必须按以下顺序从上到下排列：

```
<template>
<script setup lang="ts">
<script lang="ts">         （仅当需要 Options API 副作用时，如 inheritAttrs）
<style lang="scss">        （非 scoped 全局样式）
<style scoped lang="scss"> （scoped 局部样式）
```

**原因：** 统一 `<template>` → `<script>` → `<style>` 顺序，符合 Vue 官方风格指南推荐，也便于团队成员快速定位模板结构和逻辑代码。

**注意事项：**

- `<script>` 和 `<script setup>` 同时存在时，`<script setup>` 在前（主要逻辑），`<script>` 在后（副作用/Options API 配置）
- 无 scoped 的 `<style>` 在前（全局/基础样式），`<style scoped>` 在后（局部/覆盖样式）
- 如果原文件顺序不符合，**必须调整**为上述顺序，不可跳过
- 调整块顺序的同时保持每个块内部代码不变

### Vue 3 SFC `<script setup>` 内部排序

```
1. 依赖导入          // ==================== 依赖导入 ====================
2. Store 实例        // ==================== Store 实例 ====================
3. Props & Emits     // ==================== Props & Emits ====================
4. 模板引用          // ==================== 模板引用 ====================
5. 响应式状态        // ==================== 响应式状态 ====================
6. 计算属性          // ==================== 计算属性 ====================
7. 方法              // ==================== 方法 ====================
8. 监听器            // ==================== 监听器 ====================
9. 生命周期          // ==================== 生命周期 ====================
```

### Pinia Store（`defineStore`）

```
1. 依赖导入          // ==================== 依赖导入 ====================
2. 响应式状态        // ==================== 响应式状态 ====================
3. 内部状态（非响应式） // ==================== 内部状态 ====================
4. 注册/注销         // ==================== 注册 / 注销 ====================
5. 核心操作          // ==================== 核心操作 ====================
6. 状态重置/工具     // ==================== 状态重置 ====================
7. 对外暴露          // ==================== 对外暴露 ====================
```

### 通用排序原则

- **公开函数在前，私有函数在后**
- **被依赖方在前，依赖方在后**（例如 `sendToIframe` 在 `sendAfterReady` 前面）
- **同类元素按调用链/数据流排序**，不是字母序
- **`onMounted` 在 `onUnmounted` 前**

## 步骤 5：检查与清理

1. 移除未使用的导入（ESLint `no-unused-vars` 会报的部分）
2. 移除死代码：声明但从未使用的变量/函数
3. 合并重复逻辑：如果排序后发现有相同功能的分散代码，就近合并
4. 验证 Vue 编译器宏（`defineProps`/`defineEmits`/`withDefaults`）未被显式导入
5. 验证 `ref(null)` 是否需要显式泛型 `<HTMLIFrameElement | null>` 等

## 步骤 6：汇总报告

所有文件处理完成后，输出分类汇总报告：

```
📊 整理代码完毕

✅ 已整理 (N个)：
  - src/pages/chat/index.vue   (新增 5 条注释，重新排序)
  - src/stores/modules/chat.ts  (新增 3 条注释，移除 1 个未使用导入)

⏭️ 已跳过 (N个)：
  - src/styles/theme.scss       (应使用 mxy-organize-scss 技能)
  - src/utils/helper.ts         (已整理良好，无需改动)
  - src/business/README.md      (不支持的扩展名)
  - src/business/scbmPay/Sign.vue  (存在 Vue2 混搭语法，建议先运行 mxy-upgrade-vue3)

❌ 失败 (N个)：
  - src/components/Broken.vue    (无法解析：编译宏导入冲突)
```

**汇总规则**：

- ✅ **已整理**：实际修改了代码注释/排序的文件
- ⏭️ **已跳过**：校验阶段或执行中发现无需/无法处理的文件，必须注明跳过原因
- ❌ **失败**：处理过程中遇到不可恢复的错误的文件，必须注明错误原因

## 约束

- **不改变任何运行时行为** — 仅调整顺序和注释，不改逻辑
- **不新增依赖**
- **保留当前文件的所有 import 路径**（如 `@/stores/modules/chat.ts` vs `@/stores` 风格不一时不统一，保持文件原有风格）
- 如果单个文件已有完整的排序和注释体系，跳过该文件，在汇总报告中标记"已整理良好，无需改动"
- **批量模式下顺序执行**，不并行修改多个文件（避免多个文件的 git diff 交织）
- 文件数超过 15 个时通过 question 确认后再开始
