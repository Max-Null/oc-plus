# 设计-文档

## 核心原则

设计方案和原型文档是**互为镜像**的两份文档，必须同步创建、同步更新。设计方案面对技术团队（前后端/测试），原型文档面对产品/测试/UI。它们的映射关系：

| 设计方案章节 | 原型文档对应章节 | 同步规则 |
|------------|-----------------|---------|
| 用户故事 | 用户场景 | 一一对应，描述视角不同 |
| 关键设计决策 | 核心规则表 | 决策→规则，技术语言→产品语言 |
| 时序图 | 交互流程 | 同一流程两端视角 |
| 异常流程 | 异常分支 | 完全对应 |
| 测试要点 | 测试要点 | 设计方案偏后端，原型偏前端用户视角 |
| 风险与边界 | 边界与约束 | 一一对应 |

> 章节名称匹配，不依赖章节编号（模板可能调整顺序）。

**硬约束**：除非用户明确说"只改设计"或"只改原型"，否则每次操作同时处理两份文档。

## 工作流程

### 第一步：确认模式和文件

向用户确认两件事（不要猜）：

1. **功能名称**是什么？文件名约定为 `[功能名]-设计方案.md` 和 `[功能名]-原型文档.md`
2. **是新功能还是更新已有文档？**

如果用户不确定功能名，根据对话上下文推断一个并让用户确认。

检查当前项目文档目录。先搜后问，不要假设文件名。

> 必须通过 `question` 工具与用户确认，不可跳过。

**注意**：
- 两个确认问题必须分两次独立调用，禁止合并为一次
- 候选文件不足 2 个时，用"是这个" / "不是，手动指定"两个选项

1. 用功能名关键词在 `doc/设计/` 和 `doc/原型/`（兼容旧 `design/`）下搜索匹配文件：
   - 精确匹配到一个 → **必须调用 `question` 工具**确认"找到 `[文件名]`，是这个吗？"，选项：{"label": "是这个", "description": "确认使用该文件"} / {"label": "不是，手动指定", "description": "让我手动输入正确的文件名"}
   - 匹配到多个 → **必须调用 `question` 工具**列出候选文件供用户选择。候选文件 ≥2 个时直接列；候选文件 =1 个时降级为"精确匹配到一个"分支；候选文件 =0 个时走创建模式
   - 无匹配 → **创建模式**，按 `[功能名]-设计方案.md` / `[功能名]-原型文档.md` 命名
2. **第一次 question 完成后**，再发起**第二次独立 `question` 工具调用**确认：**是新功能还是更新已有文档？**（反向确认，防止搜索遗漏），选项：{"label": "新功能", "description": "创建全新的设计方案和原型文档"} / {"label": "更新已有文档", "description": "在现有文档基础上增量修改"}

### 第二步：加载模板

模板加载优先级：
1. **项目模板**：`doc/_模板_设计方案.md` / `doc/_模板_原型文档.md`，或旧路径 `design/_模板_*.md`（如果存在）
2. **内置模板**：`references/模板-设计方案.md` 和 `references/模板-原型文档.md`（始终可用）

### 第三步：按模式处理

#### 创建模式

1. 先和用户对话，收集以下信息（不用一口气问完，按文档章节顺序逐步确认）：
   - 需求背景：现状是什么？解决什么问题？用户故事有哪些？
   - 架构决策：涉及哪些模块？关键设计选择是什么？为什么这样选？
   - 技术方案：前后端各改哪些文件？API 怎么设计？数据库变不变？
   - 场景和原型：核心用户场景是什么？页面怎么变？
2. 生成设计方案，填充所有章节
3. 立即同步生成原型文档（不要等用户说"再写原型"）
4. 两份文档的版本号都设为 `v1.0`，填写变更记录初始行

注意：
- 如果用户已经聊了很多需求细节，直接提取，不要重复问
- 如果用户提供了代码改动，读取代码来反推设计决策
- 如果某些信息确实不确定，标注 `[待确认]` 并继续，不要卡住

#### 更新模式

1. 读取两份现有文档，获取当前版本号
2. 判断变更范围（三个来源交叉验证，取并集）：
   - 有代码改动 → `git diff` 看变更内容
   - 无代码改动 → 直接问用户"这次要更新哪些部分？"（可能是纯文档修正、补充测试用例等）
   - 对话上下文中用户提到的改动点
3. **增量更新**：只修改受影响的章节，不要重写整篇文档
4. 版本号递增（v1.0→v1.1），变更记录追加新行
5. 两份文档的变更记录同步更新——设计方案和原型文档各自记录自己的变更
6. 被更新的章节在原型文档中的对应章节也必须检查是否需要同步

### 第四步：输出

输出到当前项目的文档目录。优先匹配项目已有结构：
- 存在 `doc/设计/` 和 `doc/原型/` → `doc/设计/[功能名]-设计方案.md` + `doc/原型/[功能名]-原型文档.md`
- 存在 `design/`（旧结构）→ 保持 `design/` 目录
- 两者都不存在 → 先创建 `doc/设计/` `doc/原型/` `doc/计划/` `doc/知识/` 四个子目录（各含 README.md），再输出文档。创建目录是硬前置，不可跳过直接写文件

输出完成后简要告知用户：
- 文件路径
- 版本号
- 关键章节摘要（1-2 句话）
- 标注了 `[待确认]` 的地方

## 图表规范

- **流程图/时序图**：统一用 mermaid 语法（`flowchart TD` / `sequenceDiagram`），禁止 ASCII 字符画
- **页面原型**：ASCII 示意图（直观感受布局）+ 结构化表格（精确描述元素/条件/交互），两者搭配，不可只有其一
- mermaid 语法注意事项：禁止自闭合 `<br/>`，用括号或逗号替代；禁止 `&lt;`/`&gt;` HTML 实体，用双引号包裹直接写尖括号

## 特殊场景

### 逆向模式（代码已写完，补文档）

用户说"代码已经写了，补一下文档"时，不能仅看 diff，必须理解原系统上下文。

**第一步：建立全局认知**

1. 如果用户提供了分支名或 commit 范围 → `git diff` 精确锁定改动文件
2. 如果用户只说"代码写了" → 让用户指定改动范围（分支/commit/文件列表），不猜测
3. 读取改动的代码文件，同时读取它们**调用的已有类/组件**（如改动依赖的 Entity、已有 Service、公共工具）
4. 如有已关联的同类功能文档（`doc/设计/` 下已有方案），先读一遍，理解系统的设计惯例

**第二步：从代码反推设计**

对每个改动文件，回答以下问题再写入文档：

| 从代码中看什么 | 对应设计方案章节 |
|--------------|-----------------|
| DTO 新增了哪些字段、排除了哪些字段？为什么？ | 3.1 DTO + 2.2 决策表 |
| Controller 新增了什么端点？有没有校验/权限/事务边界？ | 3.2 Controller + 身份绑定机制 |
| Service 做了什么？事务边界在哪？涉及几张表？ | 3.4 Service 实现 + 五、数据库变更 |
| 前端页面改了哪些？模板条件渲染逻辑是什么？ | 四、前端设计 |
| 错误处理和边界条件 | 7.2 异常流程 + 九、风险与边界 |

**第三步：推导设计决策（最容易漏）**

代码告诉你"做了什么"，必须反推"为什么这样做"：
- 看到 DTO 不含某字段 → 推断"不可变更字段"，写入决策表
- 看到 `@Transactional` + CorpPerson 同步 → 推断"双表事务保护"，写入决策表
- 看到 OCR 结束后恢复原始证件号 → 推断"编辑模式证件号比对策略"，写入决策表
- 看到 `eq(User::getUsername, username)` → 推断"token 身份绑定防越权"，写入安全机制

**第四步：生成文档**

按创建模式生成，版本号 `v1.0`（首次文档化）。生成后向用户逐章确认关键推断是否正确——逆向结论有不确定性，必须经用户验证，不可直接定稿。

### 仅更新其中之一
用户明确说"只改方案"或"只改原型"时，可以单独操作。但必须提醒："另一份文档可能也有需要同步的地方，是否一起检查？"
