请严格执行以下步骤，不得跳过任何一步。每步完成后简要告知用户当前进度。

## 步骤 0：识别项目类型

执行以下命令判断技术栈：

```bash
ls pom.xml build.gradle build.gradle.kts 2>/dev/null && echo "JAVA" || true
ls requirements.txt pyproject.toml setup.py Pipfile 2>/dev/null && echo "PYTHON" || true
ls package.json 2>/dev/null && echo "NODE" || true
```

检查改动文件后缀：

```bash
git diff --name-only HEAD 2>/dev/null || git diff --name-only --staged
```

向用户确认识别结果，然后继续。

## 步骤 1：获取改动信息

```bash
git status
git diff --staged
git diff
git branch --show-current
```

若无任何改动，告知用户并终止。

## 步骤 2：审查

审查分两轮：**第一轮找问题，第二轮找更好的方案**。每轮关注不同维度。

### 2.1 防御型审查：这段代码会出什么问题？

不逐条打勾。带着以下 **4 个问题** 去读 diff，发现什么记什么：

| # | 问题 | 关注的信号 |
|---|------|-----------|
| D1 | **会不会崩？** | 空值访问、未处理异常、资源泄漏、竞态条件、类型不匹配 |
| D2 | **会不会被攻击？** | 用户输入未校验、敏感信息泄露（日志/返回值中打印 token/key/密码）、注入风险（SQL/命令/模板） |
| D3 | **会不会卡或漏？** | N+1 查询、大事务套 IO、内存泄漏（未清理的定时器/监听器）、不合理的 deep watch |
| D4 | **后来的人看得懂吗？** | 命名与行为不一致、注释过时或解释 WHAT 而非 WHY、魔法值无常量 |

> 技术栈背景知识（Java 的 NPE/事务、Python 的 GIL/可变默认参数、Vue3 的响应式丢失/异步间隙）LLM 应自行调用，**不依赖清单逐项回忆**。

### 2.2 进攻型审查：有没有更好的做法？

| # | 问题 | 关注的信号 |
|---|------|-----------|
| O1 | **设计方向对吗？** | 实现是否符合需求意图？有没有"功能都做了但方案跑偏"的情况？ |
| O2 | **是不是过度工程？** | 为一个简单需求引入了不必要的抽象层、设计模式、泛型——正确但太复杂 |
| O3 | **变更涟漪** | 改了 A 的方法签名 → B/C/D 没报错但语义已经变了。diff 内看不到，需要联想 |
| O4 | **出问题能排查吗？** | 异常被吞了、错误信息是 `e.toString()`、日志没有上下文——不崩但查不出来 |
| O5 | **回滚安全吗？** | 如果这个改动需要 revert，有没有隐式依赖（DB 迁移 / 配置项 / 接口契约变更）会让回滚炸掉？ |
| O6 | **有没有更简单的方案？** | 被跳过的、更直接的做法是什么？为什么没选？ |

### 2.3 跨边界扫描（硬约束）

如果改动涉及以下场景，**必须执行**：

**模板-脚本一致性**：`<script>` 中删除的函数/变量/import → grep 全工作区确认 `<template>` 中无残留引用（`@click="deletedFn"` 等）。遗漏 → 运行时 `undefined is not a function`。

**Emit/Props 同步**：子组件新增/删除的 emit/prop → grep 所有父组件确认同步更新。

**定时器清理**：新增的 `setTimeout`/`setInterval`/`addEventListener` → 确认返回值已捕获，`onUnmounted` 中对称清理。

**内容文件（.md）特殊审查**：若改动涉及 Agent 定义（`agents/*.md`），检查是否有 LLM 自身认知盲区的规则——要求 LLM 判断自己的认知状态（"不确定时"、"记忆模糊时"）的规则，LLM 无法执行，应改为输出模式触发。

### 审查输出格式

每个发现标注：**严重程度**（高：必崩/安全漏洞/LLM 必犯的认知错误；中：潜在问题；低：风格建议）、**文件:行号**、**问题描述**、**修复建议**。格式自由，不强制表格。

## 步骤 3：汇总审查结果

若发现问题，用 `question` 工具询问用户处理方式：

| 选项 | 标签 | 描述 |
|------|------|------|
| 1 | 修复全部 | 修复高、中、低所有严重度问题 |
| 2 | 修复中高 | 修复高、中严重度问题 |
| 3 | 仅修复高危 | 只修复高严重度 |
| 4 | 跳过修复 | 不修复，直接生成提交信息 |

若零问题，跳过步骤 4-5，直接进入步骤 6。

## 步骤 4：修复问题

- 只修复审查发现的问题，**不改变原有功能逻辑**
- 优先复用项目已有的工具函数、hooks 和编码模式
- 如果某个修复有不确定性，标注"需人工确认"并跳过

## 步骤 5：二次审查

对修复后的完整改动进行回归检查：

1. **删除符号扫描**：`git diff` 中所有被删除的函数/变量/import 名 → grep 全工作区确认无残留引用
2. **修复有效性**：对照步骤 3 汇总表，验证每项修复是否真正解决了原始问题
3. **副作用检查**：修复是否引入了新的空值风险、改变了函数签名/返回值类型、影响了其他调用方

若发现新问题，回到步骤 4。最多循环 2 次。

> 二次审查通过后，用 `question` 工具询问"是否继续提交？"（继续提交 / 中止）

## 步骤 6：生成提交信息

格式：

```
<type>(<scope>): <简短描述>

变更内容：
- <原始改动要点>

审查修复：
- <修复项>

待后续优化：
- <跳过的低严重度问题（如有）>
```

type：feat / fix / refactor / style / docs / chore / perf
scope：根据改动文件推断的业务模块名

用 `question` 工具询问：仅提交 / 提交并推送 / 修改信息

## 步骤 7：提交并推送

```bash
git add -A
# 将 commit message 写入 .git/COMMIT_MSG_TMP
git commit -F .git/COMMIT_MSG_TMP
rm .git/COMMIT_MSG_TMP
git push origin $BRANCH
```

> 禁止 `git commit -m` 传多行消息——跨 Shell 引号转义不可靠。必须用 `-F` 临时文件。

## 步骤 8：总结

用中文汇总：项目类型、提交 hash、分支、改动文件数、审查发现数、修复数、推送结果。
