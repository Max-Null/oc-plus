# 分形记忆权重分层 — 设计说明

> 版本：v1.0 | 2026-07-21 | 基于 A-MAC(2603.04549) + RAMPART(2606.04628) + Outermind 优先级模型
> 
> **验证依据**：三篇论文全文已读（2026-07），关键结论原文引用见各子节；Outermind 参考 GitHub 开源实现。非局部代码推测。

## 一、现状与缺陷

当前分形的知识注入链路：

```
记忆块文件 → parseMeta() 解析元数据 → mergeBlocksAndTriggers() 合并去重
  → 关键词匹配打分(relevance) → mtime 排序 → MAX_KNOWLEDGE_INJECT=5 硬截断
  → 统一位置 push 到 system prompt 尾部
```

**四个缺陷**：

| # | 缺陷 | 影响 |
|---|------|------|
| 1 | **无优先级** | 核心约束(如"禁止 webfetch 访问 github")和软偏好(如"文档目录结构")平等竞争 5 个坑位 |
| 2 | **无衰减** | 记忆一旦写入永不调整——低频记忆和高频记忆权重相同 |
| 3 | **等分截断** | 不管命中质量高低，统一 cut top-5。如果有 10 条高相关记忆，后 5 条被暴力丢弃 |
| 4 | **无位置优化** | RAMPART 实验证实：记忆块位置显著影响成功率，第 7 块是悬崖。相关块单独 push 比混合效果好数倍 |

## 二、目标

在不引入外部依赖、不改变文件格式兼容性的前提下，给分形记忆注入加入**确定性**的权重和分层控制。

## 三、核心设计

### 3.1 记忆块元数据扩展

在现有元数据行的基础上，新增两个可选字段（缺省值保证向后兼容）：

```markdown
<!-- type: knowledge -->
<!-- label: github-content-access -->
<!-- status: auto -->
<!-- description: GitHub | MCP | github_ | webfetch | github.com -->
<!-- priority: 95 -->          ← 新增：1-100，缺省 50
<!-- category: constraint -->  ← 新增：constraint | reference | preference，缺省 reference
```

| 字段 | 类型 | 缺省 | 说明 |
|------|------|------|------|
| `priority` | 1-100 | 50 | 权重，决定注入位置和截断保护 |
| `category` | 枚举 | `reference` | 内容类型先验（A-MAC 结论：最强的单因子） |

**category 取值与行为差异**：

| category | 含义 | 示例 | 注入策略 |
|----------|------|------|---------|
| `constraint` | 硬约束（违反即崩溃） | github-content-access、mermaid-syntax | 不衰减、不截断保护、必须注入 |
| `reference` | 参考知识（忘了也能工作） | vue3-coding-conventions、scss-coding-style | 正常衰减、正常截断 |
| `preference` | 软偏好（可覆盖） | 文档目录结构、git 提交格式 | 加速衰减、低优先级截断 |

### 3.2 排序算法

不再仅按 relevance 排序，改为加权融合：

```
score = relevance × W_relevance + priority × W_priority + recency × W_recency
```

| 权重 | 建议值 | 含义 |
|------|--------|------|
| W_relevance | 0.40 | 关键词命中数 / 最大命中数 |
| W_priority | 0.50 | priority / 100 |
| W_recency | 0.10 | e^(-days_since_write / 30) |

其中 `relevance` 不再用绝对命中数，改用**归一化命中率**（当前命中数 / 该轮所有记忆中最大命中数），防止某一轮所有记忆都低命中时硬截断。

### 3.3 四档分层注入

借鉴 Outermind，按 score 分四档，不同档用不同 prompt 格式：

| 档位 | 阈值 | system prompt 前缀 | 截断保护 |
|------|------|--------------------|---------|
| **HIGH** | score ≥ 0.8 或 priority ≥ 90 | `🔴 HIGH PRIORITY` | 不被截断，不参与 MAX_KNOWLEDGE_INJECT 计数 |
| **IMPORTANT** | 0.6 ≤ score < 0.8 | `🟡 IMPORTANT` | 正常计数 |
| **OPERATIONAL** | 0.3 ≤ score < 0.6 | `🟢 OPERATIONAL` | 正常计数 |
| **GENERAL** | score < 0.3 | 无前缀 | 优先被截断 |

**注入总预算**：`MAX_KNOWLEDGE_INJECT` 从固定值改为**动态预算** = `min(5, 上下文剩余 / 500)`。上下文紧张时自动缩减，防止记忆注入加剧 token 压力。

### 3.4 块位置优化（分组推送）

RAMPART 实验的核心发现：**相关内容放到同一块 push 中，成功率远高于分散 push**。相邻且共享 ≥2 个 description 关键词的记忆块合并为一个 system message 推送。

```
❌ 当前：
output.system.push("\n知识A\n")
output.system.push("\n知识B\n")
output.system.push("\n知识C\n")

✅ 优化后：
output.system.push("\n🔴 HIGH PRIORITY\n知识A\n知识B\n")
output.system.push("\n🟢 OPERATIONAL\n知识C\n")
```

### 3.5 PageOut 衰减

每轮未被召回的记忆按 category 衰减：

| category | 衰减策略 |
|----------|---------|
| `constraint` | 永不衰减 |
| `reference` | 连续 10 轮未命中 → priority -5；最低降至 30 |
| `preference` | 连续 5 轮未命中 → priority -10；最低降至 10 |

衰减后的 priority 回写到记忆块文件的 `<!-- priority: N -->` 行。写入前做 30 秒防抖——避免同一轮多次衰减写盘。

## 五、验证方案

| 验证点 | 方法 | 预期 |
|--------|------|------|
| 向后兼容 | 缺 priority 的旧格式记忆仍正常加载，缺省值 50 | 不影响现有记忆 |
| 分层正确性 | 话术触发 github 相关关键词，debug.log 确认 github-content-access 在 HIGH 档 | 单个 HIGH 记忆在顶部 |
| 截断行为 | 写 10 个 OPERATIONAL 记忆，触发单关键词，确认仅注入 top-5 | 未被截断保护的不超预算 |
| 分类反馈 | 调 LLM API 让分析新的习惯，确认输出的 JSON 含 priority 字段 | 新记忆附 priority |
| 衰减 | 连续不触发某记忆 5 轮，读取文件确认 priority 下降 | priority 被回写 |
