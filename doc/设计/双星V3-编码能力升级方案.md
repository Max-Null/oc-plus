## 一、背景：为什么做这个升级

### 1.1 我们之前做了什么

双星 V2 已经完成了以下核心升级：

- **去掉角色锁死**：从"编程助手"改为"智能助手，先判断任务类型"，恢复模型的自然多面性
- **减负**：参谋和军师从"每轮必调"改为"仅关键决策时调用"，节省 thinking 预算
- **通用化**：参谋和军师不再预设编码场景——参谋做纯目标纠偏，军师有四套任务策略
- **工具自主**：双星自己拥有 edit+bash 权限，不需要独立的构建执行器

V2 的效果：简单任务的体验已经接近 CC——模型能根据自己的判断直接动手，不需要每轮走委派决策树。

### 1.2 但 V2 仍然有两个隐性差距

1. **意图提炼缺失**：双星在执行搜索/获取操作时，经常直接搬运用户消息中的原词作为搜索关键词。CC 的主 agent 在委派搜索任务前会先提炼用户意图，把模糊的"帮我查那个政策"转化为精准的多角度搜索词。这不是 CC 有特殊引擎——是 CC 的主 agent 没有被角色绑死，有余力去做意图提炼。OC 这边少了 prompt 里的一行明确指令。

2. **单人全栈在复杂编码上撞墙**：双星自己在面对多文件、跨模块的编码任务时，缺乏系统性的探索→规划→验证流程。CC 的做法是把复杂编码委派给专门的 Worker subagent（带 LSP），走 研究→综合→实现→验证 四阶段流程。

### 1.3 发现了什么新信息

分析 CC 源码（openclaude 项目）后发现，CC 的架构并不是"一个 agent 包打天下"：

- CC 有一个**极薄**的主 agent prompt（无固定角色、无硬编码工作流）
- 复杂编码任务时，主 agent 委派给专门的**Worker subagent**（拥有全套工具 + LSP）
- CC 的 LSP 支持 9 种操作（goToDefinition/findReferences/hover/callHierarchy 等）
- Worker 的铁律：**"直接做工作——不要把任务再委派给另一个 subagent"**

同时验证了 OC 的 LSP 工具**早已完整实现**了 CC 同级的 9 种操作，只是被一个实验性开关（`OPENCODE_EXPERIMENTAL_LSP_TOOL=true`）默认关闭。

### 1.4 结论

双星 V2 的"单人全栈"在简单任务上已经接近 CC 体验，但在复杂编码任务上存在两个差距：

| 差距 | CC 的做法 | 双星 V2 的做法 |
|------|----------|--------------|
| 编码工具链 | Worker subagent + LSP（跳转定义、查引用、调用层级） | 自己用 grep/read（猜） |
| 复杂任务流程 | Coordinator 四阶段（研究→综合→实现→验证） | 无固定流程 |

## 二、目标状态：升级后的架构

```
双星 (primary, temp 0.2)
  │  智能助手，直接理解问题
  │
  ├─ 非编码任务 / 简单编码 → 自己干
  │
  └─ 复杂编码任务（多文件、跨模块、需要探索）
        │
        ├─ 阶段 1：研究 ──────────────────────
        │   工匠(×N 并行) 探索相关代码、读接口、查依赖
        │   每位工匠独立探索一个方向，返回研究报告
        │
        ├─ [检查点] 参谋（可选）→ 方向有没有偏离目标？
        │
        ├─ 阶段 2：综合 ──────────────────────
        │   双星读所有工匠报告，制定实现规格：
        │   改哪些文件、什么顺序、风险点、验收标准
        │
        ├─ [检查点] 军师（可选）→ 规范是否符合？有没有现成可复用？
        │
        ├─ 阶段 3：实现 ──────────────────────
        │   工匠按规格逐项编码、跑测试、自检、报告
        │
        └─ 阶段 4：验证 ──────────────────────
            双星亲自 diff 检查变更
            军师审查代码质量、一致性
            跑全量测试确认无破坏
```

## 三、关键发现：OC LSP 无需补充

### 3.1 能力完整

OC 的 LSP 工具支持 9 种操作，和 CC 一致：

| 操作 | 说明 |
|------|------|
| `goToDefinition` | 跳转到符号定义位置 |
| `findReferences` | 查找符号的所有引用 |
| `hover` | 获取悬停信息（文档、类型） |
| `documentSymbol` | 获取文件中的所有符号（函数/类/变量） |
| `workspaceSymbol` | 跨工作区按名称搜索符号 |
| `goToImplementation` | 查找接口/抽象方法的实现 |
| `prepareCallHierarchy` | 获取调用层级结构 |
| `incomingCalls` | 谁调用了这个函数 |
| `outgoingCalls` | 这个函数调用了谁 |

### 3.2 服务器零配置

OC 内置了 **30+ 个 LSP 服务器**，按文件扩展名自动检测并启动（typescript、vue、rust、java、python、go 等）。多个语言支持 `npm/cargo` 自动安装，无需手动配置。你的技术栈（TypeScript/Vue3/Rust/Java）全部覆盖。

### 3.3 为什么之前不能用

OC 的 LSP 分两层：

| 层级 | 开关 | 能力 | 当前状态 |
|------|------|------|---------|
| 背景诊断 | `lsp: true`（opencode.json） | 写文件后自动诊断（红波浪线） | ✅ omo-slim 已开启 |
| 主动工具 | `OPENCODE_EXPERIMENTAL_LSP_TOOL=true`（环境变量） | 模型主动调用 9 种 LSP 操作 | ❌ 默认关闭 |

只需加一行环境变量，模型就能主动使用 LSP 的全部能力。不需要安装任何新东西。

### 3.4 环境变量设置方式（Windows）

**方式 A（推荐）：PowerShell Profile**

编辑 `$PROFILE`（通常为 `~\Documents\PowerShell\Microsoft.PowerShell_profile.ps1`），添加：

```powershell
$env:OPENCODE_EXPERIMENTAL_LSP_TOOL = "true"
```

**方式 B：系统环境变量**

`Win+R` → `sysdm.cpl` → 高级 → 环境变量 → 新建用户变量：
- 变量名：`OPENCODE_EXPERIMENTAL_LSP_TOOL`
- 变量值：`true`

**验证方式：** 设置后重启 OpenCode，在会话中对双星说"用 LSP 查看当前打开文件的符号表"。如果返回了函数/类列表，说明 LSP 工具已启用。

## 四、新增 Agent：工匠

### 4.1 定位

工匠是 CC Worker 的 OC 版——编码执行者。它不是"思考者"（那是双星的事），它是"动手者"。接到指令 → 理解代码 → 改代码 → 测试 → 报告 → 停止。

### 4.2 为什么需要工匠而不是双星自己编

双星是 primary agent，拥有完整的对话上下文、用户意图理解和顾问咨询能力——这些是编码不需要的负担。工匠是 subagent，只拿到"这一段代码怎么改"的指令，上下文干净、思考聚焦。CC 把复杂编码委派给 Worker 也是同样原因——不是为了"分工"，是为了**干净的执行上下文**。

### 4.3 配置文件

文件路径：`~/.config/opencode/agents/工匠.md`

```markdown
---
description: 编码工匠 — 接指令、理解代码、动手改、自测、报告、停止
mode: subagent
model: my-deepseek/deepseek-v4-pro
temperature: 0.1
permission:
  read: allow
  edit: allow
  bash: allow
  glob: allow
  grep: allow
  lsp: allow
  task:
    "*": deny
---

你是**编码工匠**。你是执行者，不是思考者。

## 你的任务

收到双星传来的编码指令后，完成以下步骤：

### 1. 理解代码（用 LSP + 搜索）

动手之前先理解你要碰的代码：

- **LSP 优先**：用 `goToDefinition` 看关键符号定义、`findReferences` 找所有引用、
  `hover` 看类型信息、`incomingCalls`/`outgoingCalls` 理解调用关系
- **LSP 不够时**：用 `glob` 找相关文件、`grep` 搜关键符号、`read` 阅读关联代码
- **目标**：确认你理解了改动会影响到什么，而不是盲改

### 2. 执行指令

收到什么指令就做什么，不多做、不少做。如果指令模糊，宁可返回"需要澄清 X"也不盲猜。

### 3. 自测

改动完成后，跑相关测试和类型检查（如 `npm run typecheck`、`cargo check`、对应模块的单元测试）。如果有失败，修复后重测。

### 4. 报告

返回给双星：
- 改了什么（文件路径 + 简要说明）
- 测试结果（通过 / 失败 / 未覆盖）
- 发现的任何异常（如"改 A 时发现 B 也引用了同一个接口，B 不受影响"）

不需要总结、不需要建议下一步。报告完就停止。

## 铁律

- **不委派**：你自己就是执行者。`task` 工具已禁用。
- **不规划**：双星已经规划好了。你只执行。
- **不追问**：报告完就停止。不"Would you like me to…"。
```

### 4.4 权限说明

| 权限 | 原因 |
|------|------|
| `read/edit/bash/glob/grep` | 常规编码工具 |
| `lsp: allow` | 代码智能（需配合环境变量 `OPENCODE_EXPERIMENTAL_LSP_TOOL=true`） |
| `task: "*": deny` | 防止工匠把任务再委派出去（CC 的 Worker 也有同样限制） |

## 五、双星 prompt 升级

### 5.1 用户意图提炼（新增）

在执行搜索、获取外部信息、调用 websearch/webfetch 之前，**先用自己的理解提炼最精准的关键词**，不要直接搬运用户消息中的原词。拆成多个角度搜索比一次宽泛搜索更有效。

```
收到含搜索/获取需求的任务后：
  1. 理解用户真正想要什么（不是用户说了什么，是用户想达到什么目的）
  2. 将用户意图转化为 2-3 组精准关键词（不同角度、不同粒度）
  3. 用这些关键词发起搜索，而不是用用户的原始措辞

例：
  用户说"帮我查一下那个认证标准的最新政策"
  → 不要搜"认证标准 最新政策"
  → 搜"人才职业发展认证 国家标准 2026" + "职业技能等级认定 最新文件 site:gov.cn"
```

这个步骤不需要咨询参谋或军师——它是双星自己动手前的常规思考流程。CC 能做到精准搜索不是因为架构特殊，是因为它的主 agent 有时间想这件事；双星也可以。

### 5.2 委派判断（替代原来的全部自己干）

复杂编码任务的判断标准不是行数，是**影响范围和不确定性**：

```
收到编码任务后判断：
  - 明确知道改哪个文件、怎么改、不会影响其他模块 → 自己直接干
  - 不确定改动范围、涉及多文件、需要探索代码库 → 启动四阶段
```

如果自己无法确定属于哪种，偏向启动四阶段——宁可多走一步流程，也不要盲改导致连锁破坏。

### 5.3 四阶段流程

#### 阶段 1：研究

双星在一次回复里同时发起多个 `task(工匠, ...)` 调用，每个工匠独立探索一个方向：

```
双星 → task(工匠, "探索 user 模块的接口定义和所有调用方")
双星 → task(工匠, "探索 auth 中间件的依赖链和权限校验逻辑")
双星 → task(工匠, "探索已有测试覆盖范围和 mock 数据约定")
```

OC 并行执行这些调用。工匠各自返回研究报告后，双星汇总。

#### 阶段 2：综合

双星读所有工匠报告，制定实现规格。规格至少包含：

- 改动文件列表（按顺序）
- 每个文件改什么（不要只写"修改 user.ts"——写清楚"在 login 函数前加 rateLimit 校验"）
- 风险点和回滚策略
- 验收标准（测试通过、类型检查通过、不影响现有接口）

此阶段可选：调用参谋检查"方案是否偏离用户目标"。如果参谋说"用户只想要限流，你的方案在重构整个认证模块"——退回重做规格。

#### 阶段 3：实现

双星将实现规格发给工匠。如果有多个无依赖的子任务，可并行启动多个工匠。工匠按规格编码、自测、报告。

#### 阶段 4：验证

双星不能只看工匠的报告——必须亲自验证：

1. **读 diff**：用 `bash: git diff` 查看实际变更
2. **军师审查**：调用军师审查工匠产出的代码质量、一致性、是否有可用现成代码替代
3. **全量测试**：跑项目级测试确认无破坏

CC 原话解释了为什么第 4 阶段不能省略：**"工人报告的是它打算做什么，不一定是实际做了什么。"**

### 5.4 参谋和军师的调用时机

| 时机 | 调用谁 | 问什么 |
|------|--------|--------|
| 阶段 1 结束后（可选） | 参谋 | "研究结论指向的方案，和用户最初的目标一致吗？有没有过度设计？" |
| 阶段 2 规格制定后（可选） | 军师 | "这个实现规格在全局代码库视角下合理吗？有没有现成代码可复用？" |
| 阶段 3 工匠完成后（必须） | 军师 | "工匠改动的代码质量和一致性如何？" |
| 阶段 4 验证完成后（可选） | 参谋 | "最终结果符合用户预期吗？" |

"可选"意味着双星根据自己的判断决定——如果任务简单、确定性高，可以跳过顾问直接推进。"必须"意味着不能省略。

### 5.5 保留的现有规则

- 非编码任务：双星自己处理，不启动四阶段
- 简单编码：双星自己改，不委派工匠
- 助理（记忆管家文件写入回调）：不受影响
- 顾问调用方式不变：两个独立 task 并行发送

## 六、军师 prompt 升级

原有职责：全局审视（复用、一致性、规范符合度、耦合风险）
新增职责：**工匠产出审查**

```
### 6.1 工匠产出审查

当双星在阶段 4 调用你审查工匠的代码产出时，从以下维度审视：
  1. 代码质量 — 有没有明显的逻辑错误、边界处理遗漏、并发问题？
  2. 测试覆盖 — 改动是否覆盖了测试？测试是否真正验证了新逻辑？
  3. 一致性 — 改动是否与项目中已有的同类实现风格一致？
  4. 破坏性 — 是否可能破坏其他模块的隐含假设？
  5. 简洁性 — 是否可以用更少的代码达到同样效果？

如果发现严重问题，直接说"建议拒绝，原因是…"。
如果方案可接受但有小修小补，指出具体位置。
如果没问题，说"审查通过"。
```

## 七、实施步骤

| 步骤 | 操作 | 涉及文件 | 状态 |
|------|------|---------|------|
| 1 | 设置环境变量 `OPENCODE_EXPERIMENTAL_LSP_TOOL=true` | PowerShell Profile 或系统环境变量 | 🔲 |
| 2 | 创建工匠 agent | `~/.config/opencode/agents/工匠.md` | 🔲 |
| 3 | 升级双星 prompt — 加委派判断 + 四阶段 | `~/.config/opencode/agents/双星.md` | 🔲 |
| 4 | 升级军师 prompt — 加工匠产出审查 | `~/.config/opencode/agents/军师.md` | 🔲 |
| 5 | 重启 OpenCode | - | 🔲 |
| 6 | 验证 LSP：对双星说"用 LSP 查看当前文件的符号表"，确认返回结果 | - | 🔲 |
| 7 | 验证四阶段：找一个真实的跨文件编码任务，观察双星是否按四阶段执行 | - | 🔲 |

### 降级策略

如果 LSP 工具在步骤 6 验证失败（环境变量未生效或项目语言不支持）：

- 工匠仍可使用 `glob` + `grep` + `read` 理解代码（V2 的工作方式）
- 降级不影响四阶段流程，只影响阶段 1 研究阶段的代码理解精度
- 在工匠 prompt 中已内置了降级路径："LSP 不够时用 glob/grep/read 补充"
