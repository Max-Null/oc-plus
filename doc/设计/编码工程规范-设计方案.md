# 编码工程规范 — 设计方案

> V1.3 | 2026-07-19 | 四审修订：补充参谋角色和介入场景

## 一、设计目标

oc-plus 体系目前有两层编码保障：

| 层 | 内容 | 形态 |
|---|------|------|
| AGENTS.md | 全局编码约束（命名、注释、SCSS、Vue3 约定） | 静态文本，每轮注入 system prompt |
| 分形 Guardian | 五条触发线 | 规则引擎 + LLM 语义判断 |

这两层覆盖了**规范约定**和**行为纠偏**，但缺少**执行层方法论**。Matt Pocock 的 skills 库填补这个空白：

| 方法论 | 解决什么问题 | 嵌入目标 |
|--------|------------|---------|
| 深模块设计 | 什么是一个好接口？ | 双星阶段 2（制定规格时决策） |
| 红绿重构 | 改代码前怎么保证不破坏？ | 双星阶段 2 决策 → 转化为工匠具体指令 |
| 结构化调试 | 面对 bug 先做什么？ | 双星（遇到 bug 时的操作流程） |
| 提交前自查 | 改完代码怎么审查？ | 军师（审查基线） |

### 核心理念

**方法论嵌入在决策层，不在执行层。** 双星决定"是否需要 TDD、是否需要深模块审查"，然后转化为工匠的具体指令——工匠不需要理解方法论，只需要执行具体指令。这符合 oc-plus 现有的"双星决策→工匠执行"架构。

## 二、四个核心规范

### 2.1 深模块设计

**核心原则**：每加代码时问三问——

1. **接口够小吗？** — 方法数、参数复杂度能否再减
2. **调用方能不关心内部吗？** — 复杂度是藏在实现里，还是泄露到接口
3. **删除时复杂度会怎样？** — 删掉这个模块，复杂度是消失还是散落

**关键术语**：模块、接口（非 TypeScript 关键字）、深度、缝、局部性

**反例**：浅模块 = 大接口 + 薄实现（每个方法 2-3 行转发）；深模块 = 2-3 方法 + 复杂内部逻辑。

### 2.2 红绿重构

**核心原则**：改代码前先写失败测试（红），再通过（绿）。测试只在预确认的缝上写。

**反模式**：水平切片（批量写测试再批量实现）、实现耦合（mock 内部协作者）、同义反复（`expect(add(a,b)).toBe(a+b)`）。

### 2.3 结构化调试

**六个阶段**：

```
Phase 1 → 建立反馈环（核心，占 90% 精力）
Phase 2 → 复现 + 精简
Phase 3 → 假设列表（3-5 条，可证伪）
Phase 4 → 单变量探测
Phase 5 → 修复 + 回归测试
Phase 6 → 清理 + 复盘
```

**紧反馈环四属性**：能变红（驱动真实 bug 路径）、确定性（每次一致）、速度（秒级）、agent 可运行。

**构建反馈环 10 法**（按优先）：失败测试 → curl/HTTP → CLI+fixture → 无头浏览器 → 回放 trace → 一次性 harness → fuzz 循环 → 二分 → 差异循环 → HITL bash。

**铁律**：反馈环建成前禁止开始假设。

### 2.4 提交前自查

**双轴审查**：规范轴（代码符合规范？）←→ 规格轴（实现了需求？）——并行独立。

**12 异味基线**：神秘命名、重复代码、依恋情结、数据泥团、基本类型偏执、重复 switch、霰弹式修改、发散式变化、过度 generality、消息链、中间人、拒绝遗赠。

## 三、架构决策

### 3.1 职责分配：方法论在决策层，不在执行层

| Agent | 拥有的方法论 | 角色 |
|-------|-----------|------|
| **工匠** | 无（只有硬护栏：类型检查 + 自测 + 不吞异常） | 执行者——接到具体指令就做 |
| **双星** | 深模块设计 + 红绿重构 + 结构化调试 + 12 异味（通过调军师） | 决策者——决定用哪个方法论、转化为工匠指令 |
| **军师** | 12 异味审查基线（双轴审查） | 审查者——独立核查 |
| **参谋** | 目标锚定——防止方法论凌驾于需求之上 | 纠偏者——双星加了 TDD+深模块后，是否偏离用户原始意图？ |

> **设计原则**：工匠的核心理念是"执行者，不是思考者"。深模块三问和 12 异味自查是**设计审查行为**，与工匠身份冲突。方法论应放在决策层（双星），由双星在阶段 2 决定"这个任务需要 TDD 吗？需要深模块三问吗？"，然后转化为工匠的具体指令。

### 3.2 为什么嵌入 agent prompt 而非 blocks/

方法论是 agent 身份的一部分（"双星在阶段 2 决定是否需要红绿重构"），不是外部知识块（"用户习惯每次保存前 format"）。嵌入 prompt 随 agent 启动即生效。

### 3.3 硬护栏 vs 软护栏（修订后）

| 护栏 | 所在位置 | 性质 |
|------|---------|------|
| 类型检查通过、自测通过、不吞异常 | 工匠 prompt（硬护栏） | 执行层——具体可验证 |
| TDD 决策、深模块三问 | 双星阶段 2（检查项） | 决策层——双星决定后写入工匠指令 |
| 12 异味 | 军师 prompt（审查基线） | 审查层——独立核查 |

原有的"软护栏默认可被覆盖"概念取消——不是 LLM 理解不了优先级，而是**不该让执行者做决策**。双星决定要什么，工匠执行什么，比"默认遵守但可被覆盖"更清晰。

### 3.4 结构化调试的载体

参谋指出：V1.1 中结构化调试**没有任何 agent 内化**——仅靠触发线 2 的滑窗检测"你缺反馈环"，无法补救"你不会建反馈环"。

**V1.2 修正**：结构化调试 Phase 1-6 完整嵌入双星 prompt——双星在「遇到 bug / 连续修改不收敛」场景下，按 Phase 1→Phase 6 流程操作。触发线 2 的滑窗检测作为触发信号（提醒"你可能缺反馈环"），双星收到提醒后切入结构化调试模式。

### 3.5 触发线 2 扩展（认清延迟）

OC 1.18.3 实测：`chat.message` 同轮注入不生效，只能靠 `system.transform`（下一轮才生效）。

触发线 2 扩展后：在滑窗检测到「连续 edit 但无 bash」→ 通过 `system.transform` 在**下一轮**注入提醒。有延迟，但比完全不检测好。

---

### 3.6 参谋的角色：防止方法论凌驾于需求

**问题**：双星在阶段 2 做了深模块决策和 TDD 决策后，存在一个风险——为了方法论正确而偏离用户原始需求。例如用户只要"加一个下拉选项"，双星却套上了深模块三问 + TDD 全流程。

**参谋的介入场景**：

| 时机 | 参谋检查什么 |
|------|-----------|
| 双星阶段 2 制定规格后（含编码规范决策） | 深模块/TDD 决策是否让方案过度复杂？用户原始目标是否被方法论稀释？ |
| 遇到 Bug 的 Phase 3（假设列表形成后） | 假设是否聚焦于用户描述的症状？还是被技术细节带偏？ |
| 重大架构决策时（已有） | 方向锚定（不变） |

**决策规则**：参谋在编码规范场景下不是每次都调，仅在以下条件之一触发：
- 双星在阶段 2 勾选了多个编码规范检查项（≥3 个）
- 用户需求很简单但方案开始增加方法论开销
- 方案和原始需求之间出现了"中间层"（为了TDD设计测试、为了深模块设计接口——但用户没要这些）

## 四、agent 改造详情

### 4.1 工匠.md（基本不变）

现有工匠 prompt 已经隐含了硬护栏（步骤 3"跑类型检查和相关测试，失败则修复后重测"；铁律"不多做、不少做"）。不需要新增任何软护栏。

**唯一补充**：「不吞异常」——catch 块必须有处理逻辑（至少 console.error + 上下文信息）。

### 4.2 双星.md（核心变更）

双星是编码规范的核心载体，变更集中在两处：

#### 4.2.1 阶段 2 扩展

```markdown
**阶段 2 — 综合：** 读报告，制定规格：

1. 改哪些文件、什么顺序
2. 每个修改点的测试缝——在哪层接口验证（单元/集成/e2e），未确认缝标记高风险
3. 是否需要深模块审查（涉及"新建模块/重构接口/设计组件"→问三问）
4. 是否需要红绿重构（涉及"改代码/修bug/加功能"→在缝上先写失败测试）
5. 风险点、验收标准
```

#### 4.2.2 新增「遇到 Bug 时」

```markdown
## 遇到 Bug 时：结构化调试

当收到 bug 报告 或 分形触发线 2 警告 或 连续修改不收敛时，按以下流程操作：

**Phase 1 — 建立反馈环（核心，占 90% 精力）**
构建一个能变红、确定性、秒级的测试/脚本。按优先级：
1. 失败测试（单元/集成/e2e，在能触及 bug 的缝上）
2. curl/HTTP 脚本打运行中的 dev server
3. CLI + fixture 输入
4. 无头浏览器脚本（Playwright）
5. 一次性 harness（最小子系统 + 单函数调用）

⚠️ 反馈环建成前，禁止开始假设。

**Phase 2 — 复现 + 精简**：最小的复现步骤
**Phase 3 — 假设列表**：3-5 条可证伪假设（"如果 X 则 Y，否则 Z"）
**Phase 4 — 单变量探测**：每条假设单独验证
**Phase 5 — 修复 + 回归测试**：最小改动 + 1 个 commit
**Phase 6 — 清理 + 复盘**：grep `DEBUG-` 无残留，复盘写入记忆
```

### 4.3 军师.md（扩展审查基线）

在"审查工匠产出"部分替换为双轴审查 + 12 异味基线：

```markdown
## 审查工匠产出

双轴独立审查，互不污染：

**规范轴** — 以下 12 异味出现 ≥2 个 → 建议拒绝：
| 神秘命名 | 重复代码 | 依恋情结 | 数据泥团 | 基本类型偏执 | 重复 switch |
| 霰弹式修改 | 发散式变化 | 过度 generality | 消息链 | 中间人 | 拒绝遗赠 |

**规格轴** — 对照双星阶段 2 的规格，确认实现覆盖了所有需求点和测试缝。

报告格式：只说差异，不写表扬。严重问题说"建议拒绝"，小修小补指具体位置，通过说"审查通过"。
```

## 五、实现路径

### Phase 1：agent prompt 改造（当前）

| 步骤 | 操作 | 改动量 |
|------|------|--------|
| 1 | 工匠.md：仅补充"不吞异常" | 1 行 |
| 2 | 双星.md：阶段 2 扩展 + 新增「遇到 Bug 时」 | ~30 行 |
| 3 | 军师.md：「审查工匠产出」替换为双轴 + 12 异味 | ~15 行 |

### Phase 2：触发线 2 扩展（短期）

纯规则引擎：滑窗内连续 edit（无 bash 穿插）→ 通过 system.transform 下一轮注入提醒。

### Phase 3：自主学习（中期）

分形模式 A 分析事件日志 → LLM 检测规范违反频率 → blocks/ 调优建议。

### 不做什么

- 不把 Matt 原版 skill 直接复制
- 不新增 sub-agent
- 不给工匠加软护栏（方法论留在决策层）
- 不建独立注入周期
