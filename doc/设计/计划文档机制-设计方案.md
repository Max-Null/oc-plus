# 计划文档机制 — 设计方案

> 对标 Claude Code Plan Mode 的核心价值（持久化计划 → 防止上下文漂移），基于 oc-plus 已有四阶段流程改造。
>
> 创建时间：2026-07-22 | 状态：设计中

## 一、背景

CC Plan Mode 的工作方式：
1. 用户输入复杂任务 → CC 进入 Plan Mode（状态机切换、权限降级为只读）
2. 5 阶段工作流：理解 → 设计 → 审查 → 编写计划 → 退出审批
3. Plan 文件存储到 `~/.claude/plans/{slug}.md`
4. 退出 Plan Mode 后，系统注入「根据 plan 执行」的提醒
5. 后续执行中，plan 文件作为「上下文锚点」——即使聊天内容被压缩，plan 仍然存在

双星天然有四阶段流程，阶段 2「综合 → 制定规格」已经做了 CC Plan Mode 做的事——但规格仅存在于聊天中，没有持久化。一旦上下文压缩，计划丢失，后续步骤容易漂移。

## 二、方案概述

**不复制 CC 的状态机**（OC 没有原生 EnterPlanMode/ExitPlanMode 工具），而是利用现有能力实现等价效果：

| CC Plan Mode | oc-plus 等价方案 |
|---|---|
| 状态机（Enter/ExitPlanMode 工具） | 无状态机——双星四阶段天然分离了「计划」和「实现」 |
| 权限降级为只读 | 无需额外降级——双星已有「先对齐再动手」纪律约束 |
| Plan 文件存储 | `~/.config/opencode/plans/{日期}-{HHMM}-{slug}.md`（分钟级时间戳防重名） |
| 执行中 plan 注入 | 分形 system.transform 每轮自动注入活跃计划摘要 |
| 退出 Plan Mode 审批 | 用户直接在聊天中确认（双星输出规格后等待确认） |

## 四、计划文档格式

```markdown
# {任务标题}
> 创建时间：YYYY-MM-DD HH:MM | 状态：执行中 / 已完成 | sessionId: ses_xxx

## 需求理解
[对用户需求的解读]

## 设计方案
[技术方案概述、关键决策和理由]

## 涉及文件
| 文件 | 改动类型 | 说明 |
|------|---------|------|

## 执行步骤
- [ ] 1. ...
- [ ] 2. ...
- [ ] 3. ...

## 进度
当前步骤：2/5 | 最后更新：YYYY-MM-DD HH:MM

## 验收标准
- [ ] ...
```

> **V2 变更**：执行步骤改为 checkbox 格式，增加「进度」行。分形注入摘要时统计 `[x]` 数量，注入格式变为 `**auth 模块重构**（2026-07-22-auth-refactor.md · 2/5 完成）：重构 auth.ts...`。

## 五、计划摘要注入机制

分形 system.transform 每轮检测 `~/.config/opencode/plans/`：
- 跳过已标记「状态：已完成」的文件
- 读取 `.md` 文件，提取标题 + 执行步骤 checkbox 统计 + 第一段需求描述作为摘要
- 注入到 system prompt 中，格式：

```
## 📋 当前计划
- **auth 模块重构**（2026-07-22-auth-refactor.md · 2/5 完成）：重构 auth.ts 中间件...
```

**不受 NUDGE_INTERVAL 限制**——这是导航信息，每轮都需要。

## 六、与待办列表的关系

| | 待办列表（todowrite） | 计划文档（plans/） |
|---|---|---|
| 内容 | 执行清单，随进展动态变化 | 完整方案，静态锚点不变 |
| 生命周期 | 单次会话，可能被压缩 | 跨轮持久，分形主动注入 |
| 作用 | 追踪"现在做到哪了" | 锚定"最初要做什么" |
| 更新者 | 双星每步操作后更新 | 仅在阶段 2 创建，完成后标记状态 |

## 七、触发判据

双星判断是否需要写入计划文档（而非仅在聊天中讨论）的条件：
- 涉及 3+ 个文件的修改
- 新增模块/组件/接口
- 跨模块重构
- 用户显式说"先计划"、"先设计方案"

简单任务（单文件小改动、加注释、格式化等）不触发计划文档写入。

## 八、GUI 提取约定

未来 GUI 右侧面板需要展示三类结构化数据，各自数据源如下：

| GUI 面板 | 数据源 | 提取方式 |
|---------|--------|---------|
| **计划文档** | `~/.config/opencode/plans/*.md` | 文件系统直接读取，过滤已完成状态，按 mtime 排序。路径固定不可变，即使用户要求输出到工作区也先在 plans/ 写一份 |
| **待办列表** | 消息流 `type: "tool"`, `tool: "todowrite"` | 取最新一条 todowrite call 的 `state.input.todos` |
| **修改文件统计** | 消息流 `type: "tool"`, `tool: "edit"/"write"` | 遍历当前会话 tool parts，对 `args.filePath` 去重统计 |

**关键约定**：计划文件**只能写入 `~/.config/opencode/plans/`**，路径不可变。这是 GUI 定位计划文件的唯一锚点——

- 如果双星把计划写在项目目录下，GUI 无法区分它和普通文件
- 如果用户要求"输出到工作区"，双星 **先在 plans/ 写一份，再额外复制到工作区**

## 九、验收标准

- [ ] 双星在复杂任务阶段 2 后将规格写入 `~/.config/opencode/plans/` 目录（路径固定）
- [ ] 分形每轮注入活跃计划的摘要到 system prompt
- [ ] deploy.mjs 能创建 plans 目录
- [ ] 计划文档机制与待办列表（todowrite）不冲突，各司其职
- [ ] 执行步骤为 checkbox 格式，分形注入摘要时显示 `2/5 完成`
- [ ] 跨会话中断后，分形可恢复 plan 状态（用户说「继续」时注入精确进度）
- [ ] 阶段 4 结束后，双星自动将 plan 标记为已完成

## 十、V2 升级计划

### 10.1 升级 1：步骤级 checkbox 进度

**问题**：当前 plan 模板的「执行步骤」是纯文本列表，分形无法感知进度，摘要只能说「这个 plan 还在执行」。

**方案**：
- 计划文档格式中执行步骤改为 `- [ ] 1. ...` / `- [x] 1. ...` 的 checkbox 格式
- 分形 `getActivePlanSummaries()` 统计 `[x]` 和 `[ ]` 数量，计算完成比例
- 注入格式从 `**重构 auth**（file.md）：描述` 升级为 `**重构 auth**（file.md · 2/5 完成）：描述`
- 双星每完成一个步骤后更新对应 checkbox

### 10.2 升级 2：跨会话状态跟踪

**问题**：当前没有跨会话连续性。用户说「继续上次的任务」，双星只知道有活跃 plan，但不知道做到第几步了。OMO 用 `boulder.json` 跟踪 active_plan + session_ids + 进度，我们缺失此能力。

**方案**：
- 分形在 `getActivePlanSummaries()` 读 plan 文件的同时，将状态写入 `~/.config/opencode/plans/.active.json`
- `.active.json` 结构：

```json
{
  "activePlans": [
    {
      "file": "2026-07-22-1430-auth-refactor.md",
      "title": "auth 模块重构",
      "sessionId": "ses_abc123",
      "progress": "2/5",
      "lastCompletedStep": "修改 auth.ts 中间件",
      "lastActive": "2026-07-22T14:45:00Z"
    }
  ]
}
```

- 双星 prompt 增加「继续上次任务」处理：收到"继续"/"接着做"时，读 `.active.json` 获取精确进度
- 分形 core-rules 增加跨会话恢复规则：新会话开始时自动提示活跃 plan

### 10.3 升级 3：阶段 4 自动标记完成 + 防呆规则

**问题**：①阶段 4 结束后，双星可能忘记将 plan 标记为已完成，分形会一直注入过期计划；②缺少「修改 plans/ 文件前必须用户确认」的防呆约束。

**方案**：
- 双星 prompt 阶段 4 末尾增加一步：「将 plans/ 中对应文件的状态改为已完成」
- 双星 prompt 增加防呆规则：「修改 plans/ 目录下的文件前，必须先向用户说明原因并等待确认」
- 分形 `getActivePlanSummaries()` 已有跳过「已完成」状态的逻辑，无需改动
