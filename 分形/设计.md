# 记忆管家 — 赛博分身设计文档

> V1.x 见 [archive/设计-V1.md](archive/设计-V1.md) | V2.0 见 [archive/设计-V2.md](archive/设计-V2.md)
> 当前版本：V3.5 | 2026-07-22

## 一、设计目标

构建一个 **Guardian Agent（赛博分身）**——持续观察用户在 opencode 中的行为，在多个场景自动介入提醒，让主 agent 做出更优决策。

### 核心闭环

```
观察 → 行为模式匹配 → LLM 语义判断 → 注入 user 角色消息 → 主 agent 响应
```

与双星系统独立——赛博分身是后台观察 + 适时介入的进程，不是规划节点。

### 当前定位：五条触发线覆盖 agent 决策失误核心场景 + 编码规范联动

V2.0 只做了一件事：文件写入 → glob 匹配 → 提醒。V3.0 扩展为三条触发线。V3.1 新增触发线 4（主动联网查证），V3.2 新增触发线 5（提交后知识提取），V3.3 扩展触发线 2（无反馈环检测），V3.4 新增可配阈值、关键词精准注入、默认行为指令，V3.5 新增 pending 习惯双路除重机制。

### 设计决策与原理

每一个机制的选择都有具体的踩坑背景和权衡。

#### 1. 子目录隔离（`lib/prompts.ts`）— 为什么不能把工具模块放 plugins/ 根目录？

**背景**：OC 1.18 的自动发现用 Glob 模式 `{plugin,plugins}/*.{ts,js}` 扫描插件目录。`*` 只匹配一级文件，**每个匹配到的 .ts 文件都被当成插件处理**。`prompts.ts` 是 LLM prompt 工具模块，不是插件——放在根目录会被 OC 强行加载，报 `Plugin export is not a function` 错误。

**为什么不扔掉 prompts.ts 内联到 fractal.ts？** 两个文件职责不同：`fractal.ts` 是 OC 插件入口（hook 注册 + 事件处理），`prompts.ts` 是 LLM 分析提示词模板。分离后可以独立迭代。

**为什么选择子目录而非其他方案？**
- ❌ npm 包发布：本地开发太重，每次改 prompt 都需要 build + link
- ❌ `file://` URL：OC 1.18 不再支持 `file://` 格式的本地插件加载
- ✅ 子目录隔离：零成本，Glob 的 `*` 天然不匹配，无需改动 OC 配置

#### 2. 注入频率控制 — 为什么 knowledge 索引不每轮都发？

**背景**：长会话中，`system.transform` 每轮都注入全部 knowledge 索引 + habits 列表 + 核心规则 + 查证规则。随着 blocks/ 积累（当前 4 条，未来可能 20+），system prompt 持续膨胀。

**为什么 ACP 做不到这个？** ACP 只管「压缩」——token 超阈值时自动裁剪旧消息。但它不区分内容优先级，可能裁掉 important 的习惯提示而保留冗余的 knowledge 索引。

**为什么选 5 轮间隔？** 纯经验值——太频繁（每 2 轮）和每轮差不多，太少（10 轮）可能导致新习惯长时间不被感知。5 轮在"不过度膨胀"和"及时感知"之间取平衡。

**什么内容永远每轮发？** 核心规则（定义 agent 身份行为）和断言提醒（紧急、需要立即纠正）。这些是 Guardian 的"存在感底线"。

#### 3. 可定制 prompt 模板 — 为什么不让用户改源码？

**背景**：分形的注入文案（核心规则、断言提醒、查证规则）硬编码在 TypeScript 源码中。用户想改一个字都需要重新编译部署。

**为什么选外部文件而非配置项？**
- ❌ `opencode.json` 配置：JSON 不适合长文本编辑，且 OC 的 config schema 不支持自定义字段
- ❌ 环境变量：长度限制、不适合多段文本
- ✅ Markdown 文件：用户熟悉的格式，可直接编辑，支持分段（`---` 分隔）和占位符替换

**模板不存在时怎么办？** 回退到内置默认值。这意味着升级 plugin 源码不会破坏已有模板——用户删除模板文件即可恢复默认。

#### 4. `/fractal` 命令系统 — 为什么要从"后台脚本"变成"可见命令"？

**背景**：分形一直是后台 Guardian——静默注入、静默检测、静默记录。用户看不到它在做什么，只能通过 `debug.log` 间接感知。

**为什么需要命令？**
- **感知**：`/fractal status` 让用户知道「这个系统在运行」
- **控制**：`/fractal pause` 让用户能临时关掉不想要的提醒（而不是被迫改代码）
- **触发**：`/fractal learn` 让用户能主动要求学习（而不是等 20 事件的阈值）

**为什么不是 Web UI 或 TUI 面板？** OC 没有原生 UI 扩展机制。命令系统是 OC 支持的最低成本用户交互通道——一行配置 + 一个 CLI 脚本即可。

#### 5. 触发线 4 分级升级 — 为什么不是一刀切？

**背景**：主 agent 凭记忆下结论（如"OC 只有 3 个 hook"）是高频错误场景。AGENTS.md 有静态规则但 LLM 天然过度自信，需要外部 Guardian 强制执行。

**为什么分级？** 一刀切有两种失败模式：
- 太温和 → LLM 忽略（"提醒过了，没事"）
- 太强硬 → 破坏体验（每次都说"你必须查！"，即使只是随口一提）

分级升级在温和与强硬之间找平衡。**核心设计**：计数器跨轮持久（写文件），连续无查证的行为累积升级，正确行为立刻降级清零。

**为什么是跨轮而非同轮？** 同轮拦截技术上不可行——LLM 输出已在生成中，`chat.message.transform` 只能改输入不能截断输出。事后检测 + 事后提醒是最优可行方案。

#### 6. 触发线 3（上下文压力）— 为什么暂停？

OC 1.17+ 已内置 ACP 上下文精简插件。触发线 3 的「token 超阈值提醒」和 ACP 的「自动压缩」是同一个问题的两种解法——ACP 从执行层解决，我们再做提醒是双重开销。**不重复造轮子**是分形的设计原则之一。

#### 7. 所有 LLM 分析调用显式禁用 thinking —— 为什么不是可选参数？

DeepSeek V4 的 thinking 模式**默认开启**（官方文档确认）。分形的 LLM 调用（事件分析、trigger 语义判断、commit 知识提取）都是简单提取任务——不需要推理链，也不需要 `reasoning_effort` 控制。

显式传 `{ "thinking": { "type": "disabled" } }` 的理由：
- 不传 → thinking 默认开启 → 多花推理 token（无意义的内部推理）→ 响应更慢
- thinking 开启时 `temperature` 参数被忽略（官方文档确认），而分析任务需要低温确定性输出
- 两处调用（`analyzeAndUpdate` / `generateTriggerMessage` / `checkAndExtractCommitKnowledge`）共享同一设计决策

**结论**：这是一个防御性设计——依赖默认值会在模型行为变化时静默退化。显式禁用保证行为确定性。

#### 8. pending 习惯除重 — 为什么不能在注入阶段只靠描述去重？

**背景**：LLM 自主学习分析（模式 A）为新发现的习惯创建 `blocks/*.md` 文件，status=pending。跨轮分析可能对同一习惯生成不同文件名的区块（如 `oc-edit-then-deploy.md` 和 `edit-before-deploy.md`，描述接近但不完全一致）。当前注入阶段（`system.transform` 第 1479-1494 行）直接遍历所有 pending blocks，无除重——导致用户反复被问同一个习惯。

**为什么纯描述去重不够？** 两个问题：
- 同一习惯的描述可能不一样（"先改源码再部署" vs "先改源码再部署，不要反过来"），`===` 比较必然漏掉
- 不同文件名 + 相似描述：`blockMap` 用 label（文件名）做 key，不同文件名天然不被合并

**为什么不在 LLM 分析阶段就做语义去重？** 这是治本——创建阶段检查已有 blocks 的描述与 LLM 提议的相似度。但语义相似度需要模型调用，且无法 100% 准确（LLM 可能判断"相似"但实际是两个独立习惯）。因此需要双路处理。

**双路方案**：
- **路 1（治标·注入层）**：在 pending 注入前对描述做关键词重叠检查——两个 pending 块的 `extractKeywords()` 结果重叠率 ≥ 50% → 只保留优先级最高的一条。解决同轮反复提示问题。
- **路 2（治本·创建层）**：在 `analyzeAndUpdate` 的 LLM prompt 中增加已有习惯的反例列表，要求 LLM 输出 `"skip"` 而非 `"create"` 当检测到语义相似的已有习惯。从源头减少重复创建。

**为什么不是 100% 去重？** 语义相似 ≠ 同一习惯。两路各用一个宽松阈值（路 1 重叠率 50%、路 2 LLM 自行判断），宁可漏掉（用户手动确认）也不合并两个独立习惯。

#### 9. blocks/triggers 分子目录存储 — 为什么不能在同一个 directory 里混放？

**背景**：V3.4 之前，所有 blocks（`auto`/`suggest`/`pending`/`knowledge`）和 triggers 都混放在各自的平级目录中（`blocks/*.md` 和 `triggers/*.md`）。主 agent 想判断哪些是 pending 待确认、哪些已自动执行，必须打开每个文件解析元数据头（`<!-- status: xxx -->`）。当前 14 个文件还好，但随记忆积累，可读性和维护成本持续上升。

**为什么不用文件名前缀？** `p_`/`a_`/`s_` 前缀需要在状态变化时重命名文件。分子目录方案中文件移入对应子目录即可，文件名保持不变——对分形的 GoMap（按 label 去重）和 PageOut 衰减（按 label 定位文件）都是透明的。

**为什么选择 `pending`/`auto`/`suggest` 三个子目录而非更多？**
- 这是分形中三种 `status` 枚举的全集。`type: knowledge` 的知识块也按 status 进入对应子目录，因为 status 才是主 agent 使用者最关心的维度。
- 避免过度嵌套：status 深度已足够，不需要再加 `type/status/` 二级目录。

**向后兼容策略**：
- **读取**：`readBlocksDir()` 和 `readTriggersDir()` 同时扫描旧 flat 结构 + 新 status 子目录，保证 14 个现有文件在迁移完成前仍可被识别
- **写入**：`resolveWritePath()` 解析新建文件内容的 status 元数据 → 写入对应子目录（新建都是 `pending` → `blocks/pending/`）
- **定位**：`findBlockFile()` / `findTriggerFile()` 先在子目录找，回退 flat，确保 update/decay 操作找到正确文件

**迁移后收益**：
- 主 agent 不再需要解析元数据头判断 status — `glob blocks/pending/*.md` 一目了然
- 子目录隔离天然防止 name collision（同一文件名在不同 status 目录互不覆盖）
- 未来可扩展：`blocks/deprecated/`、`blocks/archived/` 等，不污染主目录

## 二、记忆分层

三层路径，优先级从低到高（高覆盖低）：

| 层级 | 路径 | 说明 |
|------|------|------|
| 全局 | `~/.config/opencode/memories/` | 跨项目通用习惯 |
| 个人项目级 | `~/.config/opencode/project/<hash>/memories/` | 当前项目绑定的个人偏好 |
| 共享项目级 | `<项目>/.opencode/memories/` | git 管理的团队规范 |

每层含 `blocks/`（知识块）和 `triggers/`（触发规则），按 status 分子目录：

```
memories/
├── blocks/
│   ├── pending/       ← 待用户确认
│   ├── auto/          ← 已确认，agent 自动执行
│   └── suggest/       ← 观察中，agent 参考
├── triggers/
│   ├── pending/
│   ├── auto/
│   └── suggest/
└── triggers/          ← 旧 flat 文件（迁移前兼容）
```

## 三、赛博分身身份

**你是用户的替身**——你说"按我的习惯"，因为你说的是用户的话。

| 你是 | 你不是 |
|------|--------|
| 用户的替身（Guardian Agent） | 主 agent 的监督者 |
| 多场景行为触发器 | 规划节点 |
| 后台自动进程 | 双星的一部分 |

## 四、两种工作模式

### 模式 A：分析（后台，新会话启动时触发）

system.transform 检查 events.log 增量，≥20 条时调用 LLM API 自主学习，发现新模式后写 blocks/ + triggers/（status=pending）。

### 模式 B：回应（事件驱动，多场景触发）

V2.0 实现了文件写入 → glob 匹配的单一路径。V3.0 扩展为三条触发线。

## 五、Guardian Agent：五条触发线

```
event hook 监听 tool.execute.after / file.edited / message.updated
         ↓
┌────────┼────────────────┬──────────────────────┬──────────────────────────┐
│  触发线 1               │  触发线 2              │  触发线 3                │  触发线 4
│  文件写入匹配 trigger    │  连续无进展循环        │  上下文压力              │  主动联网查证
│  (V2.0 已有)            │  (V3.0 新增)           │  (V3.0 新增, ⏸️)         │  (V3.1 新增)
└────────┼────────────────┼──────────────────────┼──────────────────────────┘
         ↓                ↓                      ↓                         ↓
    三层漏斗         纯规则判断              纯规则判断               分级升级
  (glob→LLM→注入)  (窗口统计→注入)        (阈值→注入)          (断言检测→频率追踪→分级干预)
```

### 触发线 1：文件写入匹配 trigger（V2.0 已有）

文件写入 → glob 预筛选 → LLM 语义判断 → prompt 注入。三层漏斗架构不变。

详见 [archive/设计-V2.md](archive/设计-V2.md) 第五章。

### 触发线 2：连续无进展循环 + 无反馈环检测（V3.0 新增，V3.3 扩展）

**场景**：主 agent 反复修改代码但问题不收敛，且未调用 web_search/fetch。

**检测逻辑**（纯规则，无 LLM 调用）：

```
滑动窗口：最近 5 次 tool.execute.after
条件 A：≥3 次是 edit/write/apply_patch（在反复改代码）
条件 B：0 次是 web_search / web_fetch / mcp__github（没联网查）
条件 C：≥2 次同一文件被连续 edit（在同一个坑里跳）
→ 全部满足：触发
```

**注入消息**（纯模板，无 LLM 生成）：

```
> [记忆管家] 检测到连续 {N} 轮代码修改未收敛，且无联网查证
你是不是该查一下官方文档？之前踩过类似的坑——猜测不如查证。
```

### 触发线 4：主动联网查证（V3.1 新增·分级升级）

**场景**：主 agent 在不熟悉某 API/框架/工具时，凭借训练数据中的不准确记忆下结论，而不是先联网验证。这是 OC 官方 AGENTS.md 已明确禁止的行为（「禁止凭记忆输出以下类型的结论」），但 LLM 天然有过度自信倾向，需要外部 Guardian 强制干预。

**核心理念**：不分级的一刀切提醒效果差——一次温和提醒可能被忽略，但每次都强硬又破坏体验。分级升级在温和与强硬之间找平衡。

```
用户: "这个库支持批量删除吗？"
  ↓
LLM 生成响应: "不支持，只能单条删"  ← 断言但没调 websearch
  ↓
event hook 检测 → 信号文件写入
  ↓
下一轮 system.transform:
  首次 → 温和提醒: "你刚才说『不支持』但没有联网查证，上心点？"
  累犯 → 升级提醒: "连续 N 次断言不查证了，现在搜！"
  惯犯 → 强制指令: "回复前必须先调 websearch"
```

**检测逻辑**：

```
┌─ Pre-turn（system.transform）────────────────────────────────┐
│  注入强化版「联网查证」规则，位置在 system prompt 末尾        │
│  比 AGENTS.md 的静态规则更靠后 = 更高的 recency weight       │
│  包含：禁止输出类型清单 + 正反例（❌/✅） + 分级干预状态     │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌─ LLM 生成响应 ───────────────────────────────────────────────┐
│  响应中如果包含断言 + 本轮未调 websearch → 下次触发          │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌─ Post-turn（event hook: message.updated）─────────────────────┐
│  1. 正则匹配断言（现有 ASSERTION_RE）                         │
│  2. 检查本轮 tool call 列表中是否有 websearch/webfetch        │
│  3. 有断言 + 无联网 → 递增计数器 + 写信号文件                │
│  4. 有断言 + 有联网 → 重置计数器（本分安全）                  │
│  5. 无断言 → 计数器缓慢衰减（用户可能换了话题）               │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌─ Next-turn（system.transform）───────────────────────────────┐
│  读取信号文件 + 频率计数器 → 按等级注入：                      │
│                                                                 │
│  等级 0（count = 0）：静默，不注入                              │
│  等级 1（count = 1）：温和提醒                                 │
│    「上轮你说了『不支持』但没有联网查证——先查再下结论？」     │
│                                                                 │
│  等级 2（count = 2~3）：升级为强硬提醒                          │
│    「连续 {count} 次断言未查证了！涉及未知 API 时，             │
│     调用 websearch 比凭记忆猜强 10 倍。」                       │
│                                                                 │
│  等级 3（count ≥ 4）：警告+自动触发                             │
│    「你已连续 {count} 次断言未查证！本轮回复前，               │
│     如涉及任何 API/工具/框架能力判断，必须先调 websearch。」    │
└──────────────────────────────────────────────────────────────┘
```

**计数器管理**（`~/.config/opencode/memories/.assertion-counter.json`）：

```jsonc
{
  "count": 2,
  "lastSnippet": "不支持 XLSX 导出",
  "lastSessionId": "abc123"
}
```

- 计数跨轮、跨重启持久（写入文件）
- 有断言+无联网 → `count++`
- 有断言+有联网 → `count = 0`（本轮正确行为抵消）
- 连续 3 轮无断言 → `count = max(0, count - 1)`（衰减）
- 会话切换（sessionID 变化）→ `count = 0`（新话题重置）

**设计决策：为什么不是同轮拦截？**

同轮拦截（chat.message.transform 中检测断言后阻止响应）在技术上不可行——LLM 输出已经在生成中。可行的替代是双通道注入：
- `system.transform` 注入预防规则（事前）
- `event hook` 检测断言 → 写信号（事后）
- 下一轮 `system.transform` 读取信号，按等级施加干预（反馈闭环）

这与 ACP 的双通道架构（system + message）思路一致，但介入时机不同：ACP 是"当下太多东西，压缩掉"，分形触发线 4 是"上次你犯了错，这次严肃点"。

### 触发线 5：提交后知识提取（V3.2 新增·LLM 分析）

**场景**：每次 git commit 是积累知识的最佳时机——但手动记录容易忘。提交信息 + diff 包含了「做了什么 + 为什么做」的完整信息，LLM 可从中提取值得跨会话保留的知识。

**检测逻辑**（轮询时间戳，零 token）：

```
system.transform 读取 git log -1 最新提交 hash + 时间戳
  ↓
对比 .commit-last-check.json 中记录的上次提交时间戳
  ↓
有新提交 → 读取 git log -1 + --stat → 调用 LLM 分析
  ↓
LLM 判断是否值得记录 → 写入 blocks/
```

**为什么用轮询而非事件 hook？** OC 1.18.3 不触发 `tool.execute.before` / `tool.execute.after` 事件（实测验证），无法在 commit 后立刻捕获。`system.transform` 每轮都会执行，轮询 `git log` 时间戳是最可靠的替代方案。

**过滤规则**：
- 跳过非 git 目录
- 跳过自动提交（`/^(Merge|Bump|chore\(deps\)|\(bot\))/i`）
- LLM 判断无价值 → 跳过

### 触发线 3：上下文压力（⏸️ ACP 已覆盖）

**检测逻辑**（纯规则）：

```
监听 session.status 事件中的 token 用量
条件：当前 token 数 > 模型窗口上限 × 70%
→ 触发
```

**注入消息**（纯模板）：

```
> [记忆管家] 上下文已占用 72%，建议安装 ACP 或手动 /acp compress
命令：opencode plugin opencode-acp@latest --global
```

## 六、实现架构

### 三层漏斗（触发线 1 专用）

V2.0 架构完整保留，不做改动：

```
文件写入事件
  → 第一层：glob 预筛选（TypeScript，<1ms，零 token）
  → 第二层：LLM 语义判断（异步 HTTP API）
  → 第三层：消息注入（client.session.promptAsync）
```

### 纯规则判断（触发线 2、3、4 新增）

不需要 LLM API 调用，直接在 event hook 中统计判断 + 模板注入：

```
event hook → 滑动窗口统计 / 阈值判断 → 模板消息注入
```

延迟 <1ms，零 token 消耗。

### 断言计数器（触发线 4 专用）

```typescript
const ASSERTION_COUNTER = path.join(MEMORIES_DIR, ".assertion-counter.json");

interface AssertionState {
  count: number;        // 当前累计断言（未查证）次数
  lastSnippet: string;  // 最近一次检测到的断言片段
  lastSessionId: string;
  updatedAt: string;
}
```

与触发线 2 的滑动窗口不同——计数器需要跨轮、跨重启持久（写入 JSON 文件），因为分级升级依赖历史累积。触发线 2 的窗口只管最近 5 次，不需要持久。

## 十、上下文精简：ACP 选型

### 为什么不自研

DCP/ACP 是 OC 社区最成熟的上下文精简方案（2.3K〜19K 周下载），已经解决了上下文压缩的 90%——工具注册、模型自主压缩、保护机制、嵌套压缩。自研是重复造轮子。

### ACP 选型理由

| 对比维度 | DCP | ACP（选中） |
|---------|-----|-----------|
| 维护状态 | 开发放缓，转向 Sleev | 活跃维护（2026.7 仍在更新） |
| Bug 修复 | — | 35 个 bug fixed over DCP |
| 过度压缩 | 已知问题 | v1.10+ 修复了 nudge 频率过高导致过度压缩 |
| 自适应阈值 | 固定 token 触发 | v1.8.1+ nudge 根据模型窗口大小自适应 |
| 上下文降智 | 短会话激进裁剪 | 可通过 `minContextLimit: "45%"` 配置延迟触发 |

### 已知 Trade-off

- **短会话/小任务**：过度压缩导致 LLM 细节丢失。ACP 默认 `minContextLimit: 45%` 已缓解
- **成本**：cache miss 率上升（~80% vs 85%），但 token 节省通常覆盖
- **保护**：需配置 `protectedTools` 保护关键工具输出免于裁剪

### 分形与 ACP 协同

```
分形（判断层）：检测上下文压力 → 提醒 → 用户确认
ACP（执行层）：注册 compress 工具 → 模型自主调用 → 外部执行
```

两个层面互补：分形做"何时该裁"的决策，ACP 做"怎么裁"的执行。Guardian Agent 后续可感知 ACP 的压缩行为，做动态阈值判断（短会话不裁、长会话裁、调试模式保护关键信息）。
