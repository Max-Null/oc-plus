# 记忆管家 — 赛博分身设计文档

> V1.x 历史设计见 [archive/设计-V1.md](archive/设计-V1.md)
> 当前版本：V2.0 | 2026-07-17

## 一、设计目标

构建一个「赛博分身」——持续观察用户在 opencode 中的行为，自主发现重复模式（habits），在类似场景再次出现时主动替用户发出指令，让主 agent 执行对应的习惯动作。

### 核心闭环

```
观察 → 发现模式 → 存储 habit → 下次场景匹配 → 替用户发指令 → 执行
```

与双星系统独立——赛博分身是后台观察 + 注入建议的进程，不是规划节点。

## 二、记忆分层

三层路径，优先级从低到高（高覆盖低）：

| 层级 | 路径 | 说明 |
|------|------|------|
| 全局 | `~/.config/opencode/memories/` | 跨项目通用习惯 |
| 个人项目级 | `~/.config/opencode/project/<hash>/memories/` | 当前项目绑定的个人偏好 |
| 共享项目级 | `<项目>/.opencode/memories/` | git 管理的团队规范 |

每层含 `blocks/`（习惯描述，供参考）和 `triggers/`（执行规则，供匹配）。

### 文件格式

**block**：HTML 注释元数据 + 正文

```markdown
<!-- type: habit | knowledge -->
<!-- label: 标签名 -->
<!-- description: 简短描述 -->
<!-- confidence: high | medium | low -->
<!-- status: pending | suggest | auto -->

习惯描述...
```

**trigger**：元数据 + 触发规则（GitHub Actions 风格 schema）

```markdown
<!-- type: habit | knowledge -->
<!-- human_description: 给人看的说明 -->
<!-- confidence: high | medium | low -->
<!-- status: pending | suggest | auto -->

trigger:
  on: file_created
  match:
    - "glob模式"
  exclude:
    - "排除模式"

action:
  type: review
  focus:
    - 审查重点

message_template:
  "你刚生成了 {filename}，按我的习惯，先审查一遍。重点看：{focus}"
```

## 三、赛博分身身份

**你是用户的替身**——你说"按我的习惯"，因为你说的是用户的话。

| 你是 | 你不是 |
|------|--------|
| 用户的替身 | 主 agent 的监督者 |
| 习惯的触发者 | 规划节点 |
| 后台自动进程 | 双星的一部分 |

## 四、两种工作模式

### 模式 A：分析（后台，新会话启动时触发）

system.transform 检查 events.log 增量，≥20 条时调用 LLM API 自主学习，发现新模式后写 blocks/ + triggers/（status=pending）。

### 模式 B：回应（事件驱动，文件写入时自动触发）

V2.0 重构为三层漏斗架构（见下章）。

## 五、V2.0 回应模式：三层漏斗

V1.x 的痛点：回应模式依赖主 agent 主动调 task tool 启动 sub-agent，导致硬编码强依赖（system prompt 膨胀 25 行）+ 无差别盲打 + 上下文污染。V2.0 将回应从「主 agent 主动调」重构为「event hook 被动监听 + LLM 语义判断」。

```
文件写入事件（file.edited / tool.execute.after）
    ↓
┌──────────────────────────────────────────────────┐
│ 第一层：glob 预筛选（TypeScript，<1ms，零 token）  │
│ 事件属性提取文件路径 → 遍历 triggers/               │
│ → 匹配 status=auto + glob 规则 → 过滤 90%+ 无关写入 │
└────────────────────┬─────────────────────────────┘
                     │ 匹配到
┌────────────────────▼─────────────────────────────┐
│ 第二层：LLM 语义判断（异步 HTTP API）              │
│ prompt = 助理.md 身份定义 + 文件路径 + trigger 内容 │
│ 判断：glob 匹配 == 场景真正匹配？                   │
│ 匹配 → 第一人称提醒消息 / 不匹配 → 空，静默         │
└────────────────────┬─────────────────────────────┘
                     │ 匹配
┌────────────────────▼─────────────────────────────┐
│ 第三层：消息注入（client.session.promptAsync）     │
│ 消息以 user 角色注入，主 agent 当作用户消息处理      │
│                                                  │
│ > [记忆管家] 匹配习惯「文档审查」                   │
│ > (glob: **/*.md) | 置信度 high                   │
│                                                  │
│ 你刚生成了 plan.md，按我的习惯，你先审查一遍。       │
│ 重点看：步骤可行性和时间估算。                      │
└──────────────────────────────────────────────────┘
```

### 透明度标注

注入消息首行用 markdown 引用块标注触发来源（谁触发、哪个习惯、glob 依据、置信度）。引用块中的元信息不污染正文「替用户说话」的语义。

### LLM API prompt（第二层）

```
你是用户的赛博分身。你像用户一样思考。

当前场景：
- 文件：{filePath}
- 匹配到的习惯规则：{trigger完整内容}

你的任务：
1. 判断是否真正匹配此习惯（不只看 glob，理解上下文）
2. 匹配 → 以用户口吻生成提醒：
   > [记忆管家] 匹配习惯「{human_description}」
   > (glob: {match_globs}) | 置信度 {confidence}

   你刚生成了 {filename}，按我的习惯，先审查一遍。
   重点看：{focus_items}
3. 不匹配 → 返回空
4. 措辞用"按我的习惯"
5. 只说话，不执行

返回纯文本。
```

## 六、助理 sub-agent 角色

V2.0 中 `助理.md` 不再被 system.transform 或 event hook 自动调用。

| 之前（V1.x） | 现在（V2.0） |
|---|---|
| 强制自动调用的 sub-agent | LLM API prompt 的语义来源 |
| 回应模式的唯一入口 | 备用手动入口 |
| 写在 agent 定义文件中 | 同上，保留但不部署为自动触发 |

赛博分身从「显式 sub-agent」变为「后台自动进程」。用户看不到助理，但透明度标注保证可观测。
