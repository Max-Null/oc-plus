# 记忆管家 — 赛博分身设计文档

## 一、设计目标

构建一个「赛博分身」Agent，持续观察用户在 opencode 中的行为，自主发现重复模式（habits），在类似场景再次出现时主动替用户发出指令，让主 agent 执行对应的习惯动作。

### 核心闭环

```
观察 → 发现模式 → 存储 habit → 下次场景匹配 → 赛博分身替用户发指令 → 主 agent 执行
```

### 与双星系统的关系

- 记忆系统（赛博分身）与协作系统（双星）独立，职责正交
- 记忆系统通过 system prompt 注入增强双星系统
- 不合并为"三星"——赛博分身是一个持续观察并注入建议的后台进程，不是规划节点

## 二、记忆管家记忆分层

```
┌──────────────────────────────────────────────────────────────┐
│                     静态记忆层                                │
│  ~/.config/opencode/AGENTS.md                               │
│  行为准则 / 编码规范 / 技术栈 / 自检规则（手动维护）           │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                 动态记忆层（赛博分身管辖）                     │
│                                                              │
│  ~/.config/opencode/memories/                               │
│  ├── events.log              ← 事件原始日志                   │
│  ├── debug.log               ← 诊断日志                       │
│  ├── last-analysis.json      ← 分析进度记录                   │
│  ├── blocks/                 ← 记忆块（习惯描述）              │
│  │   ├── review-habits.md    ← 赛博分身自动维护               │
│  │   └── ...                  ← 不预定义，agent 自主扩展       │
│  └── triggers/               ← 触发指令（习惯的执行规则）       │
│      ├── auto-review.md      ← 赛博分身生成                   │
│      └── ...                  ← 每条高置信度 habit 一条         │
└──────────────────────────────────────────────────────────────┘
```

### 文件框架（Letta 风格三段式）

**block 文件**（描述习惯，供主 agent 参考）：

```markdown
<!-- label: review-habits -->
<!-- description: 用户对代码和文档质量的审查偏好 -->
<!-- confidence: high -->
<!-- confidence_reason: 跨3个会话、2个项目，每次文件生成后都会手工审查 -->
<!-- status: active -->

## 代码审查
- 用户会逐行检查逻辑漏洞、边界条件、命名规范
```

**trigger 文件**（执行规则，供 system prompt 注入）：

```markdown
<!-- label: auto-review-docs -->
<!-- description: 生成计划/设计文档后自动审查 -->
<!-- confidence: high -->
<!-- confidence_reason: 跨会话多次确认，每次生成设计类文档后必审查 -->
<!-- status: auto-execute -->

## 触发条件
- assistant 生成了 .md 文件（文件名含 计划/plan/设计/spec）
- 排除：文件名含 README

## 执行动作
- 主 agent 完成文件生成后，调用 task tool 启动赛博分身 subagent
- 赛博分身判断是否匹配已知 habit
- 匹配时返回：「你刚生成了 [文件类型]，按我的习惯，你先审查一遍吧。重点看👇...」
- 不匹配时返回：「无事，继续」

## 演进记录
- 2026-07-01 首次发现（置信度 low，仅记录不触发）
- 2026-07-08 跨会话再次确认 → 升级为 suggest
- 2026-07-13 跨项目持续出现 → 升级为 auto-execute
```

## 三、赛博分身 Agent 定义

### 3.1 基本配置

```yaml
---
description: 赛博分身 — 用户行为模式的观察者与替身
mode: subagent
model: my-deepseek/deepseek-v4-pro
temperature: 0.3
permission:
  read: allow
  edit: allow
  bash: deny
---
```

### 3.2 身份定位

你是用户的**赛博分身**。你像用户一样思考，观察用户在 opencode 中的所有操作，发现"用户反复做的事"，然后在下次类似场景出现时代替用户做出反应。

**你不是**：
- 不是主 agent 的监督者（你不是审代码的）
- 不是另一个规划节点（你不管怎么执行任务）
- 不是双星系统的左脑或右脑

**你是**：
- 用户的替身——你说"按我的习惯"，因为你说的是用户的话
- 习惯的发现者——从 events.log 中自己找出规律
- 习惯的触发者——场景匹配时，替用户发指令

### 3.3 工作模式

赛博分身有两种工作模式：

**模式 A：分析模式（后台，会话结束时触发）**

由 `session.compacting` hook 或累积 N 条事件后唤醒。

职责：
1. 读取 events.log 中的新增事件
2. 分析：有没有反复出现的操作序列？
3. 发现模式后：
   - 写入 blocks/ 目录（描述这个习惯）
   - 如果置信度 >= 建议阈值，写入 triggers/ 目录（生成执行规则）
4. 更新 last-analysis.json

**模式 B：回应模式（对话中，被主 agent 的 task tool 调用）**

职责：
1. 接收主 agent 传来的"我刚做了什么"的摘要
2. 检查：是否匹配 triggers/ 中任一触发条件？
3. 匹配 → 以用户的语气返回指令（措辞为"按我的习惯，你..."）
4. 不匹配 → 返回空或"无事，继续"

### 3.4 置信度机制

置信度由 **LLM 自主判断**，不按固定次数决定。LLM 综合分析以下维度后给出 confidence（high / medium / low）和 suggested_status：

| 维度 | 高置信度特征 | 低置信度特征 |
|------|-------------|-------------|
| 时间密度 | 同一会话内频繁出现 | 跨度几天才出现一次 |
| 跨上下文一致性 | 多个项目/多种文件类型都出现 | 仅限某个特定场景 |
| 操作紧密度 | A 操作后几乎总是紧接 B | A 和 B 之间经常有其他操作 |
| 用户主动程度 | 用户手动执行，有明确意图 | 被动触发或可能是偶然 |

confidence → suggested_status 参考映射（Plugin 代码做最终兜底映射）：

| confidence | suggested_status | 行为 |
|:--:|------|------|
| low | suggest | 只写 block 文件，不生成 trigger，不注入 system prompt |
| medium | suggest | 生成 block + trigger，注入时标记为「建议执行」 |
| high | auto | 生成 block + trigger，注入时标记为「必须自动执行」 |

最终状态由**用户确认**决定，LLM 的 confidence 和 suggested_status 均为建议。 |

### 3.5 工具权限

赛博分身在分析模式下需要：
- 读：events.log、已有 blocks/、已有 triggers/、last-analysis.json
- 写：blocks/*.md、triggers/*.md、last-analysis.json
- 调 LLM：用于分析事件序列（使用 @opencode-ai/sdk 或 fetch 调用 API）

## 四、执行流程

### 4.1 运行时流程（对话中触发）

```
用户开始使用 opencode
    ↓
system.transform 注入 blocks + triggers 到 system prompt
    ↓
主 agent 按系统指令工作（如双星系统的 双星）
    ↓
主 agent 完成一次文件操作（如生成了 plan.md）
    ↓  （system prompt 注入了例行规则）
主 agent 调用 task tool → 赛博分身 subagent（模式 B：回应模式）
    ↓
赛博分身检查 triggers/ → 匹配到 auto-review-docs
    ↓
赛博分身返回：「你刚生成了 plan.md，按我的习惯，你先审查一遍吧。
   重点看：逻辑完整性、步骤可行性、时间估算。」
    ↓  （tool result 回到对话）
主 agent 收到 tool result，像处理用户消息一样执行审查
    ↓
用户看到整个对话流程：文件生成 → 赛博分身介入 → 审查 → 结果
```

### 4.2 分析流程（会话结束后触发）

```
会话结束 / 累积 20 条以上新事件
    ↓
memories.ts 的 session.compacting hook 触发
    ↓
通过 task tool 调用赛博分身 subagent（模式 A：分析模式）
    ↓
赛博分身读取 events.log 新增事件
    ↓
分析：用户在这段时间内的操作序列有什么规律？
    ├── 没有新发现 → 更新 last-analysis.json，结束
    └── 发现了新 habit → 写入 block 文件
        └── confidence 达 medium/high → 同时写入 trigger 文件
    ↓
更新 last-analysis.json，记录分析时间戳
```

## 五、memories.ts 改造清单

基于当前 `plugins/memories.ts`（293 行），需要做以下改动：

### 5.1 system.transform 改造

注入策略按 type + status 两级分类：

| type | status | 注入标题 | 措辞风格 |
|------|--------|---------|---------|
| habit | auto | 已确认的习惯 | "你会自然地做这些——像肌肉记忆一样" |
| habit | suggest | 观察中的习惯 | "你偶尔这样做，仅供参考" |
| habit | pending | 待确认 | 提示 agent 用 question 工具逐条向用户确认 |
| knowledge | auto/suggest | 你记录的元知识 | "你之前主动要求记住的信息" |
| knowledge | pending | 同上 | 同上（用户主动记的无需确认，直接 auto） |

用户确认交互：
- 检测到 pending 的习惯 → 注入提示让 agent 用 question 工具确认
- 选项：保存为自动执行(auto) / 保存为建议执行(suggest) / 不保存(skip)
- agent 根据用户选择编辑文件 status 或删除文件

### 5.2 事件记录改造

当前 `event` hook 只记录 `message.updated`。改造后记录更多事件类型：

- `message.updated`（已有）
- `tool.call` 和 `tool.result`（如果 opencode 暴露出这些事件类型）
- 记录时附加会话 ID 和时间戳，便于赛博分身做序列分析

### 5.3 会话结束触发分析

`session.compacting` hook 中增加：

1. 检查 events.log 中自上次分析以来的新事件数
2. 如果 >= 20 条，通过 task tool 调用赛博分身（模式 A）
3. 赛博分身完成分析后，更新 blocks/ 和 triggers/

## 八、设计审查结论（2026-07-13）

### 已确认的调整

| # | 决策 |
|---|------|
| 1 | 分析模式触发时机精简为 system.transform 单一入口（compacting 助力不大，去掉） |
| 2 | 当前方案：分析模式走 plugin fetch 调 LLM（已有 Phase 2 先例），回应模式走 agent task tool |
| 3 | 分析分身活在 plugin 代码中（有写权限），回应分身独立 agent 定义（read only），权限分离 |
| 4 | 三层记忆架构：全局 / 个人项目级 / 共享项目级 |
| 5 | trigger 文件用 GitHub Actions 风格 schema（on + match(glob) + exclude），顶部加 human_description |
| 6 | 分析模式统一计数，回应模式只匹配不写 |

### 三层记忆存储路径

```
~/.config/opencode/memories/                  ← 全局习惯（跨项目）
~/.config/opencode/project/<hash>/memories/   ← 个人项目级（不跟随 git）
<项目>/.opencode/memories/                    ← 共享项目级（git 管理）
```

### 理想方案（当前不可行，记入未来升级）

若 opencode SDK 未来支持 `client.agent.invoke({ agent: "助理", ... })` 或等效 API：
- 分析模式可从 TypeScript fetch 迁移到纯 agent 定义，实现分析逻辑与代码分离
- 插件 event hook（如 file.edited）可直接启动 subagent，不需要经过注入消息→主 agent 响应的间接路径

**2026-07-17 联网验证结论**：
- OC Issue #20387 确认：`tool.execute.before/after` **不能从 hook 中 spawn sub-agent**。这是 CC agent hook 的核心能力，OC 当前缺失。
- 但有 **两种替代路径** 可实现 C（双 agent 校验）：
  1. 注册自定义 tool `memory-verify`（`tool` hook），主 agent 可调它做联网验证
  2. `tool.execute.after` + `client.session.prompt({ noReply })` 注入合成消息
- 两条路都依赖同一前提：**升级插件签名到标准 PluginInput**（拿到 `client`）
- 社区已有 `opencode-claude-hooks` 包实现 CC→OC hook 映射，证明可行

### 替代升级路径（当前可行）

利用 `client.session.prompt({ noReply: true })` + event hook：
```
event hook 监听 file.edited
  → 匹配触发条件
  → client.session.prompt({ noReply: true }) 注入消息到会话
  → 主 agent 看到注入消息后自然响应
```
优于纯 fetch 方案的点：回应逻辑可活在 agent 定义中，不需要维护两份 prompt。
